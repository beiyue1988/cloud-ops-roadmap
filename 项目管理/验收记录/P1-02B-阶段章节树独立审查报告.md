## 8. 五个阶段的人工质量审查

### 8.1 阶段 06：日志、审计与故障诊断

章节树形成日志来源、`journald`、`rsyslog`、日志轮转、审计查询、故障证据闭环的连续路径，能够从阶段 05 的服务运行基础平稳进入诊断能力。内容没有提前承担阶段 07 的安全策略设计，也没有越界到阶段 16 的集中式日志平台。

### 8.2 阶段 07：性能调优与安全加固

章节树先建立共同诊断入口，再分出性能和安全两条能力线，并在 `07.26` 收束。对 CPU、内存、磁盘、网络、安全基线、权限和审计的前置引用明确，没有提前扩展到容器平台、SRE 治理或合规体系。

### 8.3 阶段 08：自动化编程

共同基础之后形成 Shell 必需线和 Python 建议线。`08.17` 可独立支撑就业主线，`08.32` 代表完整自动化编程能力；这种双重可用结构符合已批准设计。章节粒度保持在知识章节级，没有写成脚本、实验步骤或项目实现。

### 8.4 阶段 09：Web 与 Java 服务运维

核心路径覆盖 Web 服务和 Java 服务运维并收束于 `09.15`；Tomcat、DNS、chrony、NFS、Samba 等内容作为建议掌握分支继续到 `09.25`。核心就业路径不依赖完整可选服务树，职责也没有侵入数据库、中间件或后续平台阶段。

### 8.5 阶段 10：数据库、缓存与消息系统

MySQL 与 Redis 分别建立核心能力，再由 `10.28` 收束就业主线。高可用、RabbitMQ 和 Kafka 保持建议掌握且与必需闭包隔离。阶段 11 现有骨架明确承接阶段 10，并转向自动化协作；以 `10.28` 作为稳定前置不需要隐藏依赖 `09.25` 或 `08.32`。

五个阶段的每个章节均保持一个主要学习目标，预计时间只使用 30–120 分钟的允许区间；章节名称、目标、类型、就业标签、依赖、实践锚点和理由字段具有可执行性，但没有生成正式学习正文、实验、脚本、配置、图表、故障案例或面试题。
