# P0-05 版本与来源基线设计说明

**状态：** 已批准  
**设计日期：** 2026-07-12  
**制定者：** 主会话  
**批准方案：** 分层版本基线（方案 A）  
**批准日期：** 2026-07-12  
**前置基线：** P0-04、P0-04B-R2 均已 `accepted`

## 1. 背景

项目已经在 PRD 中确定 Rocky Linux、Kubernetes、containerd、Calico、VitePress、RuoYi-Vue 和云上 GPU 等关键方向，但尚缺少一个可供后续章节、实验、项目和审查共同引用的版本入口。若各任务自行选择版本，会出现正文重复写死补丁、同一组件使用不同教学线、来源无法追溯，以及“查阅过官方资料”被误写成“已经完成环境验证”等问题。

P0-05 负责建立初始版本与来源治理基线。它既不能过早锁定全部技术栈，也不能只记录当前使用的少数组件。已批准方案将版本治理状态与验证证据状态分离：核心教学线现在锁定，后续技术在进入对应阶段前核定，历史版本只用于对比；来源核验与环境验证分别记录。

本设计取代 `2026-07-10-Phase-0-治理与骨架-实现计划.md` 中 P0-05 的简略规格。原计划中关于 VitePress 的 ADR 不再依赖未经本任务重新核验的“替代工具处于维护模式”结论，而改为比较真实可行方案的能力、维护成本和项目适配性。PRD 已批准的产品决定、版本方向和许可证边界不变。

## 2. 目标

1. 建立后续内容唯一引用的软件版本基线入口；
2. 明确 `locked`、`phase-gated`、`reference-only` 三种治理状态；
3. 明确 `source-verified`、`environment-verified`、`not-yet-verified` 三种证据状态；
4. 锁定当前核心实验线，并为未进入实施阶段的技术设置最晚核定门；
5. 建立按事实类型选择一手来源、处理来源冲突和记录动态事实的方法；
6. 用 ADR 固化 VitePress、Kubernetes 教学线和 RuoYi-Vue 项目基线；
7. 为未来升级提供变更日志、触发条件、回退与复审入口。

## 3. 非目标

- 不安装、升级或运行 Rocky Linux、Kubernetes、containerd、Calico、VitePress 或 RuoYi-Vue；
- 不把来源核验宣称为实验、集成或生产验证；
- 不为全部技术提前选择具体版本；
- 不编写知识章节、实验正文、项目脚本、配置文件或架构图；
- 不下载或提交 RuoYi-Vue 完整第三方源码；
- 不创建文档站、CI、自动更新机器人或外链监控；
- 不决定 GPU 实例族、具体 GPU 型号、地域、目标模型或 CUDA 组合；
- 不修改 PRD、AI 写作规则、许可证或已验收工具；
- 不执行 push、Pull Request、发布或 GitHub 设置变更。

## 4. 已批准方案

采用“治理状态 + 证据状态 + 更新门”的分层模型。

### 4.1 治理状态

| 状态 | 定义 | 允许内容 | 禁止解释 |
|---|---|---|---|
| `locked` | 项目已经决定教学线、固定标识或平台选择 | 固定 minor、LTS 线、tag、commit、安装方式或平台选择；补丁按策略集中维护 | 不表示所有补丁均已实际跑通，也不表示永不升级 |
| `phase-gated` | 组件在项目范围内，但具体版本必须在最晚决策门前核定 | 组件范围、最晚核定节点、核定依据和更新触发条件 | 不填写猜测版本，不允许执行任务绕过决策门 |
| `reference-only` | 只用于历史、迁移、兼容或对比说明 | 历史版本、旧基线及适用边界 | 不得作为新实验或默认教程基线 |

`locked` 锁定的是表中“锁定粒度”所指对象。Kubernetes 可以锁定 `1.36` minor，同时把当前 patch 作为动态字段；RuoYi-Vue 则同时锁定 tag 与 commit。正文不得把这两种锁定粒度混为一谈。

### 4.2 证据状态

| 状态 | 定义 | P0-05 可否写入 | 最低证据 |
|---|---|---|---|
| `source-verified` | 已由直接支持结论的一手资料核验 | 可以 | 官方文档、发布页、支持矩阵、仓库 tag/commit 或许可证文件，并记录核验日期和限制 |
| `environment-verified` | 已在明确环境中实际运行并保存验证证据 | 仅有既存验收证据时可以 | 已验收实验、测试或工具报告；必须引用环境、命令和结果 |
| `not-yet-verified` | 尚未完成足以支持版本结论的核验 | 可以 | 写明最晚验证门和所需证据，不得伪造候选版本 |

P0-05 的网络查阅只能产生 `source-verified`。唯一允许继承为 `environment-verified` 的初始项是已经通过 P0-04 验收的仓库校验器 Python 运行时；其教学用 Python 版本仍单独保持 `phase-gated`。

### 4.3 版本字段

版本基线主表至少包含以下字段：

1. 组件或组件组；
2. 用途范围；
3. 教学线或固定标识；
4. 当前来源核验版本；
5. 锁定粒度；
6. 治理状态；
7. 证据状态；
8. 选择理由；
9. 最晚核定或验证节点；
10. 官方来源；
11. 核验日期；
12. 更新触发条件。

不存在具体结论时使用“进入对应任务前核定”，不能使用空单元格、猜测数字或含义不明的“最新版本”。

## 5. 初始基线边界

### 5.1 现在锁定

| 组件 | 锁定内容 | 初始证据边界 |
|---|---|---|
| Rocky Linux | `9.8`、`x86_64`、主实验操作系统 | 官方发布与生命周期来源；不代表 VMware 和云镜像均已安装验证 |
| Kubernetes | `1.36` minor、`kubeadm` 安装方式 | 执行日重新核验当前 patch、支持状态和 EOL；当前设计核验值为 `1.36.2` |
| containerd | `2.3` LTS 教学线、CRI v1 | 执行日重新核验当前 patch；环境配置仍在 Kubernetes 实验中验证 |
| Linux cgroup | cgroup v2、`systemd` driver | 依据 Kubernetes 与 containerd 官方要求；不单独伪装成软件版本 |
| Calico | `3.32` minor | 执行日重新核验 patch，并确认官方测试矩阵仍包含 Kubernetes 1.36 |
| RuoYi-Vue | tag `v3.9.2`、commit `0e2d75c`、官方 Gitee 上游、MIT License | 固定教学业务项目；升级必须由新决策替代，不跟随 `master` |
| 仓库校验器 Python | P0-04 实际使用的 Python `3.11.15` | 可引用 P0-04 最终验收为 `environment-verified`；不等同于 Python 教学线 |

补丁版本必须以执行日的一手来源为准。若执行日在 Kubernetes 1.36.3 或其他补丁正式发布之后，基线记录新的官方 patch，但不得把尚未完成实验验证的 patch 标记为 `environment-verified`。

### 5.2 按阶段核定

版本基线必须为以下组件或组件组建立 `phase-gated` 条目，并写明最晚核定门：

| 组件或组件组 | 最晚核定门 |
|---|---|
| VMware 具体产品与版本 | 阶段 03 实验环境任务进入 `ready` 前 |
| Python 教学线及主要库 | 阶段 08 样章或实验任务进入 `ready` 前 |
| Nginx、JDK/Java 与应用运行依赖 | 阶段 09 样章或传统部署任务进入 `ready` 前 |
| MySQL、Redis、RabbitMQ/Kafka | 阶段 10 样章或传统部署任务进入 `ready` 前 |
| Git、Ansible、Terraform/OpenTofu | 阶段 11 样章或实验任务进入 `ready` 前 |
| Docker Engine、Buildx、Compose | 阶段 13 样章或容器化项目进入 `ready` 前 |
| Jenkins、GitLab CI、GitHub Actions、制品与镜像仓库 | 阶段 15 样章或流水线实验进入 `ready` 前 |
| Filebeat/Fluent Bit、Logstash、Elasticsearch、Kibana、Loki | 阶段 16 样章或可观测项目进入 `ready` 前 |
| Prometheus、Alertmanager、Grafana、OpenTelemetry、Jaeger/SkyWalking | 阶段 17 样章或可观测项目进入 `ready` 前 |
| VitePress、Node.js 与包管理器 | 文档站任务进入 `ready` 前 |
| NVIDIA 驱动、CUDA、Container Toolkit、PyTorch、vLLM 与目标模型 | 阶段 18 AI Infra 实验进入 `ready` 前 |
| 阿里云 GPU ECS 实例族、GPU、显存、地域与计费条件 | 阶段 18 AI Infra 实验进入 `ready` 前 |

阶段门条目可以记录选择标准和官方入口，但不得记录未经本任务直接核验的候选版本。RuoYi-Vue 固定 tag 中声明的依赖只能作为后续核定输入，不自动升级为整个知识库的全局版本。

### 5.3 仅供参考

Kubernetes `1.31` 必须登记为 `reference-only`，说明它是旧讨论中的废弃项目基线、不得用于新实验，并链接当前 Kubernetes ADR。其他历史版本只有在已进入正文范围或替代关系需要追溯时才加入，不建立无用途的版本收藏表。

## 6. 补丁、升级与验证策略

1. 教程正文优先引用版本基线，不在多个章节复制当前 patch；
2. 组件的 minor、LTS 线、tag 或 commit 按锁定粒度管理；
3. 安全 patch 发布后先完成来源核验，再由对应实验验证后更新证据状态；
4. 出现 EOL、官方兼容矩阵变化、关键 CVE、仓库不可用、API 弃用或实验无法复现时触发复审；
5. 涉及 minor、major、固定 tag 或架构组合变化时，必须更新或替代 ADR；
6. 只变更当前 patch 且不改变兼容边界时，可以更新基线和更新日志，但仍需验证证据；
7. 更新日志必须记录日期、组件、旧值、新值、原因、证据、关联任务或 ADR、验证状态和影响范围；
8. 版本降级或回退必须保留原因和适用边界，不能删除历史记录。

## 7. 来源治理模型

`参考资料/来源优先级.md` 不重复 PRD 的简单排行，而是按事实所有者路由来源。

| 事实类型 | 首选来源 | 交叉核验 |
|---|---|---|
| 标准、协议与格式 | 标准组织、RFC、规范发布者 | 实现项目的官方兼容说明 |
| 版本、生命周期、API 与功能 | 上游项目官方文档、Release、tag、支持页 | 官方仓库、变更日志 |
| 多组件兼容性 | 双方官方兼容矩阵或要求，采用可证明的交集 | 发行说明与后续环境验证 |
| 发行版包、路径与默认行为 | Rocky Linux 或对应发行版官方文档、仓库元数据 | 上游文档，明确发行版差异 |
| 云产品、实例、库存与价格 | 阿里云官方产品文档、价格页和带日期的控制台观察 | 地域、计费方式与实例条件 |
| 安全公告 | 上游或厂商安全公告、CVE 分配机构 | 发行版安全公告和修复版本 |
| 许可证与归属 | 材料仓库中的 LICENSE、NOTICE、tag 对应文件 | SPDX 或许可证官方说明 |
| 历史与设计动机 | 官方项目历史、设计文档、Release Notes | 经典书籍或经交叉验证的资料 |

来源记录至少包含：被支持的结论、发布者、直接 URL、适用版本或范围、核验日期、证据类型和不能证明的内容。搜索摘要、聚合下载站、培训博客和 AI 生成文本不能作为易变或关键结论的唯一来源。

来源冲突按以下顺序处理：

1. 先确认两份来源是否讨论同一产品、版本、日期和适用范围；
2. 优先采用事实所有者的直接来源；
3. 对兼容性采用双方明确支持范围的交集；
4. 比较发布日期、更新时间、Release 与文档分支；
5. 在基线中记录能够解释的差异和限制；
6. 无法解释时停止结论，将组件保持 `phase-gated` 或 `not-yet-verified`，交由主会话裁决。

## 8. ADR 设计

### 8.1 ADR-0003：VitePress 文档站

- **决定：** VitePress 是默认文档站工具，Markdown 保持单一正文源；具体 VitePress、Node.js 和包管理器版本在网站任务进入 `ready` 前核定。
- **真实选项：** VitePress；MkDocs；存在必要证据时可以补充 Docusaurus，但不能制造陪衬选项。
- **必须说明：** 技术文档定位、Front Matter、本地搜索、Markdown 适配、Node.js 工具链成本、替代方案的实际优缺点和复审触发条件。
- **禁止：** 未核验就声称替代工具停止维护；把工具选择等同于当前依赖版本已经锁定。

### 8.2 ADR-0004：Kubernetes 1.36 教学基线

- **决定：** 锁定 Kubernetes 1.36 minor，以执行日官方安全 patch 为当前来源核验值；配套采用 kubeadm、containerd 2.3 LTS、CRI v1、cgroup v2、systemd driver 和 Calico 3.32 minor。
- **真实选项：** Kubernetes 1.36；仍处于支持期的 Kubernetes 1.35。
- **历史排除：** Kubernetes 1.31 已 EOL，不构成当前可选教学基线，只记录为被放弃的旧讨论。
- **必须说明：** 支持窗口、版本偏差与兼容矩阵、补丁集中维护、升级触发、实验验证门和回退影响。

### 8.3 ADR-0005：RuoYi-Vue v3.9.2 教学基线

- **决定：** 使用官方 Gitee 仓库的 `v3.9.2` tag 和 commit `0e2d75c`；保持 MIT License；默认不提交完整第三方源码。
- **真实选项：** 固定 release tag；跟随上游 `master`。
- **必须说明：** 可复现性、更新成本、许可证与归属、依赖核定边界，以及升级必须由替代决策完成。

三份 ADR 可由 P0-05 执行会话按任务包明确授权记录为 `accepted`，因为对应选择已由用户和主会话批准。执行会话不得改变决定，只能依据最新一手证据准确记录。

## 11. 验证与验收边界

### 11.1 自动验证

1. 仓库校验器退出码为 0；
2. `git diff --check` 退出码为 0；
3. 十个白名单文件全部存在，白名单外修改为 0；
4. 基线包含三种治理状态和三种证据状态定义；
5. 所有规定的锁定组件、阶段门组件组和 Kubernetes 1.31 参考条目可检索；
6. 三份 ADR 包含模板必需结构、至少两个真实选项和直接证据；
7. 三个 README 均建立有效本地入口；
8. 仓库中不存在新增疑似凭据。

### 11.2 人工验证

1. 锁定粒度与当前 patch 没有混淆；
2. 来源核验与环境验证没有混淆；
3. 当前 patch、发布日期、支持状态和兼容矩阵与执行日官方页面一致；
4. phase-gated 条目没有猜测版本，且均有最晚核定门；
5. 来源优先级按事实类型路由，不是简单复制 PRD；
6. ADR 选项真实可行，理由和负面后果完整；
7. RuoYi-Vue 的 tag、commit、上游与许可证准确；
8. 更新日志能够支持未来追溯与回退；
9. 文档未承诺已经完成未发生的安装或集成验证；
10. 外部链接直接支持相邻结论，限制与核验日期明确。

## 12. 提交与审查策略

P0-05 执行会话形成两个提交：

1. 基线提交：只包含六个创建的基线/来源/ADR 文件和三个 README 修改；
2. 报告提交：只包含 `项目管理/执行报告/P0-05-执行报告.md`。

执行报告正文只记录第一个提交的完整哈希。第二个提交的完整哈希只在最终回复中给出，禁止为回填自身哈希产生第三个提交。禁止 amend、rebase、push、创建 Pull Request、发布或修改 GitHub 设置。

P0-05 完成后只能建议 `submitted`。主会话先做范围和事实初审，再由 P0-06 对 P0-01 至 P0-05 的 Phase 0 成果进行独立审查；P0-06 不自动替代主会话最终验收。

## 13. 停止条件

- 官方来源显示已批准教学线不再受支持或兼容矩阵不成立；
- Kubernetes、containerd、Calico 的支持范围无法形成可证明交集；
- RuoYi-Vue tag、commit、上游或许可证与批准结论不一致；
- 来源冲突无法按第 7 节解释；
- 需要选择未获批准的 minor、major、tag、平台或许可证；
- 需要白名单外写入、修改 PRD/AI 规则或改变已批准 ADR 决定；
- 只能依赖搜索摘要、第三方博客或 AI 结论支撑关键事实；
- 发现真实凭据、签名 URL 或其他敏感信息；
- 仓库校验失败且根因不在白名单内；
- 无法形成规定的两个提交或工作区不能恢复干净。
