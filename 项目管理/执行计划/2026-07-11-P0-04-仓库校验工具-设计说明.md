# P0-04 仓库校验工具设计说明

**状态：** 已批准
**设计日期：** 2026-07-11
**制定者：** 主会话
**前置基线：** P0-03 与 P0-03B 均已 `accepted`
**执行环境基线：** Python 3.11.15，仅使用标准库
**批准日期：** 2026-07-11

## 1. 背景

P0-03 已建立十类模板、占位符协议、Front Matter 字段和链接边界。后续内容任务数量增加后，如果继续依赖每个执行会话临时拼接 PowerShell 与 `rg` 命令，容易出现规则漂移、检查构造错误、错误顺序不稳定和报告数字不可复现。

P0-04 负责把适合机械判断的基础门禁固化为一个本地、跨平台、可单元测试的 Python CLI。自动校验只判断机器可确认的事实，不能替代技术、教学、安全或来源的人工审查。

本设计取代 `2026-07-10-Phase-0-治理与骨架-实现计划.md` 中 P0-04 的简略规格。主要增强点为：明确 Front Matter 子集语义、稳定错误编号、代码围栏处理、链接越界与大小写检查、基础敏感信息扫描、不少于 24 个测试，以及新增 P0-04B 独立审查。Phase 0 其他任务范围不变。

## 2. 已批准方案

采用单文件 Python 校验器与单文件单元测试：

```text
脚本资源/项目校验/validate_repository.py
脚本资源/项目校验/tests/test_validate_repository.py
脚本资源/项目校验/README.md
项目管理/执行报告/P0-04-执行报告.md
```

选择理由：

- 当前规则规模适合以纯函数组织在一个实现文件中；
- 不引入包结构、配置文件或第三方依赖；
- 执行会话和学习者可以直接运行；
- 未来规则显著增长时可以在保持 CLI 与函数契约的前提下拆包；
- 比 PowerShell 更适合开源仓库的跨平台使用和单元测试。

不选择多模块 Python 包，因为当前职责尚不足以抵消额外导入、路径和维护复杂度；不选择 PowerShell 作为主实现，因为它会弱化 Linux/macOS 可运行性。

## 3. 目标

1. 提供零第三方依赖的仓库 Markdown 基础校验入口；
2. 检查知识章节 Front Matter、重复章节编号、本地链接、正式内容占位符和明显敏感信息；
3. 用稳定规则编号、相对路径、行号和排序输出可复现错误；
4. 通过 TDD 证明正例、反例、边界和 CLI 行为；
5. 让后续任务在本地和未来 CI 中复用同一入口；
6. 不把机械校验宣传为完整 Markdown/YAML/安全或教学审查。

## 4. 非目标

- 不接入 GitHub Actions 或其他 CI；
- 不访问网络，不检查 HTTP/HTTPS 链接可达性；
- 不安装 PyYAML、Markdown 解析器或其他第三方依赖；
- 不实现完整 YAML、CommonMark 或 VitePress 解析器；
- 不自动修改或修复仓库文件；
- 不运行 VitePress 构建、Mermaid 渲染或 Draw.io 导出；
- 不校验页内锚点是否存在；
- 不校验正文技术真实性、教学质量、来源权威性或版本兼容性；
- 不创建或修改 P0-05 版本基线；
- 不把基础敏感信息扫描描述为完整的 Secret 检测方案；
- 不执行 push、Pull Request、发布或 GitHub 设置变更。

```text
项目管理/验收记录/P0-04B-独立审查报告.md
```

## 6. 架构与数据流

校验器保持单文件，但内部按职责分层：

1. **规则常量层：** 扫描目录、排除目录、必需键、允许枚举、规则编号和高置信度模式；
2. **文本预处理层：** UTF-8 读取、Front Matter 边界识别、代码围栏屏蔽和行号保持；
3. **单文件规则层：** Front Matter、链接、占位符和敏感信息；
4. **仓库聚合层：** Markdown 发现、重复 `chapter`、路径相对化、错误排序和摘要统计；
5. **CLI 层：** 参数、输出流、退出码与异常边界。

数据流：

```text
根目录
  → 确认目录与解析绝对路径
  → 发现 Markdown（排除 .git）
  → UTF-8 读取
  → 按文件适用范围运行规则
  → 聚合跨文件 chapter
  → 按路径、行号、规则编号、消息排序
  → 输出错误或成功摘要
  → 返回 1 或 0
```

任何文件读取、解析或校验异常都必须转换成稳定错误，不允许默认打印 traceback。

## 7. 公开函数契约

保留既有六个函数签名：

```python
extract_front_matter(text: str) -> dict[str, str]
validate_front_matter(path: Path, text: str) -> list[str]
validate_local_links(path: Path, text: str, root: Path) -> list[str]
validate_placeholders(path: Path, text: str) -> list[str]
validate_repository(root: Path) -> list[str]
main(argv: list[str] | None = None) -> int
```

新增一个直接可测试的规则函数：

```python
validate_sensitive_information(path: Path, text: str) -> list[str]
```

契约说明：

- `extract_front_matter` 返回顶层键到原始规范化文本的映射；标量保存标量文本，列表或 `versions` 子映射保存其缩进块文本；它不承诺返回通用 YAML 类型；
- 重复键、分隔符和行号由内部解析函数保留，`validate_front_matter` 负责把它们转换为错误；
- 单文件函数返回已经格式化的错误字符串，不写标准输出，不修改文件；
- `validate_repository` 聚合所有规则并返回确定性排序后的错误；
- `main` 是唯一负责 CLI 输入输出和退出码的函数；
- 内部辅助函数以下划线开头，不构成长期公开 API。

## 8. Markdown 与代码围栏处理

代码围栏识别同时支持反引号和波浪线，起止围栏必须使用同一字符且结束长度不得短于开始长度。屏蔽代码围栏时保留换行数量，使后续诊断行号与原文一致。

- Front Matter 只在文件第一行开始识别，不受围栏处理；
- 本地链接和占位符检查忽略代码围栏中的示例；
- 敏感信息检查不忽略代码围栏，因为凭据即使位于示例中也不能入库；
- 未闭合代码围栏屏蔽到文件末尾，不把其中示例误报为链接或占位符；本阶段不把未闭合围栏本身作为 Markdown 语法错误。

## 9. Front Matter 规则

### 9.1 适用范围

只对 `知识库/` 下非 `README.md` 的 Markdown 强制 Front Matter。其他目录不会因缺少 Front Matter 报错。

### 9.2 约定子集

解析器只支持项目知识模板实际需要的子集：

- 文件第一行和结束行使用独立的 `---`；
- 顶层 `key: value`；
- 顶层键下的缩进列表；
- `versions` 下的一层缩进映射；
- 双引号或单引号标量按文本去除外层引号；
- `[]`、`{}` 和普通标量；
- 空行与注释行。

不支持锚点、别名、标签、多行折叠标量、复杂键或更深嵌套。遇到超出子集的结构返回明确 `FM` 错误，不猜测语义。

### 9.3 必需键和取值

必需顶层键精确为：

```text
title
chapter
stage
type
difficulty
prerequisites
objectives
study_time
practice_level
keywords
versions
status
last_updated
```

规则：

- 必需键不得缺失、重复或为空；
- `prerequisites`、`objectives`、`keywords` 必须为非空列表；
- `versions` 必须为 `{}` 或至少包含一个非空键值；
- `type` 只允许 `concept`、`principle`、`service`、`platform`、`tool`、`language`、`cloud-service`、`hardware`、`methodology`；
- `status` 只允许 `draft`、`reviewed`、`verified`、`deprecated`；
- `last_updated` 必须匹配 `YYYY-MM-DD` 并能由标准库解析为真实日期；
- `chapter` 在全部受检知识章节中必须唯一；错误同时指向冲突双方。

不在 P0-04 检查 `chapter` 与文件名、目录编号或学习路线的对应关系；这些关系需要章节树稳定后另行设计。

## 10. 本地链接规则

### 10.1 扫描范围

检查仓库中除 `.git/` 外所有 Markdown 的本地链接和图片引用。支持：

- 行内链接与图片：`[文本](目标)`、`![说明](目标)`；
- 使用尖括号包裹、包含空格的目标；
- 引用式链接定义：`[标识]: 目标`。

README、模板和治理文档同样检查，因为它们是项目导航入口。

### 10.2 忽略与解析

- 忽略 `http://`、`https://`、`mailto:` 和纯 `#anchor`；
- `file.md#anchor` 或带查询参数的目标只检查文件部分；
- 对 URL 百分号编码使用标准库解码；
- 不检查外部 URL 可达性，不检查页内锚点是否存在；
- 代码围栏内链接示例不参与检查。

### 10.3 安全与跨平台

- 相对目标必须解析在仓库根目录内；越界目标即使存在也报错；
- 绝对本地路径和 Windows 盘符链接报错；
- 目标必须存在，可以是文件或目录；
- 在大小写不敏感文件系统上逐段比较实际名称，发现大小写不一致时报告 `LINK` 错误，避免在 Linux、GitHub 或 VitePress 中失效；
- 不跟随仓库外目录完成校验。

README 必须说明本阶段不是完整 CommonMark 解析器；包含嵌套括号等超出约定语法的复杂目标，应改用尖括号形式或后续解析器任务。

## 11. 正式内容占位符规则

只扫描以下目录中的 Markdown：

```text
学习路线/
技术索引/
知识库/
实验手册/
项目实战/
故障案例/
面试体系/
运维思维训练/
企业规范/
```

扫描正文时忽略代码围栏，ASCII 词不区分大小写。禁止：

```text
TODO
TBD
待补充
后续完善
稍后完善
这里填写
<必填：...>
<可选：...>
<填写...>
<替换...>
```

`TODO` 和 `TBD` 只按独立英文词匹配，避免误伤较长标识符。中文词和尖括号占位符按明确模式匹配。

`模板/`、`项目管理/`、`项目记忆/`、`版本记录/`、`参考资料/`、`脚本资源/` 和根治理文件不执行占位符规则，因为这些目录可能合法描述或测试占位符协议。正式内容不能利用排除目录规避检查；文件归属由信息架构和任务包决定。

## 12. 基础敏感信息规则

对仓库全部 Markdown 扫描高置信度模式，包含代码围栏：

- 具有固定结构的云 AccessKey；
- PEM/OpenSSH 私钥头；
- `access_key`、`secret_key`、`api_key` 等直接赋非占位值；
- 明确的云 AccessKey/Secret 组合。

模式必须排除 `<...>` 占位符、环境变量引用和明确的空值；README 必须列出覆盖范围和误报处理方式。测试凭据使用运行时字符串拼接并明确为虚构数据，不能在测试源码中保存看似可用的完整凭据。

本规则不扫描二进制文件，不替代专用 Secret 扫描器，也不证明仓库绝对没有敏感信息。

## 13. 规则编号与错误格式

规则编号前缀：

| 前缀 | 范围 |
|---|---|
| `FMxxx` | Front Matter 边界、结构、键和值 |
| `CHxxx` | 跨文件重复章节编号 |
| `LINKxxx` | 链接缺失、越界、绝对路径和大小写 |
| `PHxxx` | 正式内容占位符 |
| `SECxxx` | 高置信度敏感信息 |
| `IOxxx` | 根目录、读取和 UTF-8 解码 |

错误行格式固定为：

```text
相对/路径.md:行号: 规则编号 错误说明
```

要求：

- 仓库根错误使用路径 `.` 和行号 `0`；
- 路径统一使用 `/`；
- 错误按路径、行号、规则编号、消息排序；
- 同一事实只生成一个主错误，避免重复噪声；
- 错误说明包含实际值和期望，但不泄露完整敏感值；
- 敏感信息错误只显示类型和位置，不回显匹配内容。

## 14. CLI 设计

命令：

```powershell
python "脚本资源/项目校验/validate_repository.py" .
```

参数：

- 可选位置参数 `root`，默认当前目录；
- `--help` 由 `argparse` 提供；
- P0-04 不增加配置文件、忽略规则参数、输出格式参数或自动修复开关。

输出与退出码：

- 无错误：标准输出打印扫描 Markdown 数、知识章节数、链接数和各规则摘要，返回 `0`；
- 有错误：标准错误逐行打印排序后的错误，最后打印错误总数，返回 `1`；
- 根目录不存在、不是目录、文件读取失败或 UTF-8 解码失败均转换为 `IO` 错误并返回 `1`；
- `--help` 返回 `0`；
- 默认不显示 traceback。

同一仓库连续运行两次，诊断和摘要必须字节级一致。

## 15. 测试策略

使用 `unittest` 与 `tempfile.TemporaryDirectory`，测试通过临时仓库构造正例和反例，不修改真实项目内容。测试文件使用标准库加载校验模块，不依赖当前目录碰巧位于 `sys.path`。

测试不少于 24 个，至少覆盖：

### 15.1 Front Matter

1. 完整合法 Front Matter；
2. 缺少 Front Matter；
3. 未闭合 Front Matter；
4. 缺少必需键；
5. 重复顶层键；
6. 空标量或空列表；
7. 非法 `type`；
8. 非法 `status`；
9. 格式错误或不存在的日期；
10. 合法列表与 `versions` 子映射；
11. 超出约定 YAML 子集；
12. 两个文件重复 `chapter`。

### 15.2 链接

13. 合法相对文件和图片；
14. 外部 URL、邮件与纯锚点被忽略；
15. `file#anchor` 只检查文件；
16. 失效相对链接；
17. 越出仓库根目录；
18. 绝对路径或盘符；
19. Windows 大小写不一致；
20. 代码围栏内链接被忽略；
21. 引用式链接定义与带空格尖括号目标。

### 15.3 占位符与敏感信息

22. 正式正文占位符被拒绝；
23. 代码围栏中的占位符被忽略；
24. 排除目录中的协议示例不报错；
25. 模板批准占位符残留在正式内容时报错；
26. 高置信度虚构 Secret 被拒绝且消息不回显原值；

### 15.4 仓库和 CLI

27. 中文目录和 UTF-8 文本；
28. 无法解码文件产生 `IO` 错误；
29. 错误排序稳定；
30. CLI 成功返回 `0`；
31. CLI 失败返回 `1`；
32. 不存在根目录返回 `1` 且无 traceback；
33. 当前真实仓库校验通过。

实现可以合并密切相关的断言，但最终测试方法数不得少于 24，且上述 33 个行为必须可从测试名称或断言映射追溯。

## 16. TDD 与执行顺序

1. 读取任务包、批准设计、实施计划和 P0-03 模板基线；
2. 创建测试文件，不创建实现；
3. 运行完整 `unittest discover`，保留因实现模块缺失而失败的红灯证据；
4. 创建最小实现，使测试逐步转绿；
5. 运行完整测试，测试方法不少于 24 且全部通过；
6. 运行 `python -m py_compile`；
7. 对当前真实仓库运行 CLI，返回 `0`；
8. 连续运行两次并比较输出，结果一致；
9. 创建 README，复跑测试、CLI、链接、安全、白名单和 `git diff --check`；
10. 提交校验器、测试和 README；
11. 创建并单独提交执行报告；
12. 主会话范围初审后解锁 P0-04B；
13. 独立审查通过后，主会话最终验收。

红灯证据必须来自实际执行且失败原因确为实现缺失，不能人为写一个永远失败的断言。绿灯证据必须来自提交前的完整测试套件。

## 17. 提交边界

```text
test: add repository content validator

```

第一提交精确包含：

```text
脚本资源/项目校验/validate_repository.py
脚本资源/项目校验/tests/test_validate_repository.py
脚本资源/项目校验/README.md
```

第二提交精确包含：

```text
项目管理/执行报告/P0-04-执行报告.md
```

P0-04B 独立审查提交：

```text
docs: review repository content validator
```

它只包含：

```text
项目管理/验收记录/P0-04B-独立审查报告.md
```

所有提交均禁止未经任务包授权的 amend、rebase、push、PR、发布和 GitHub 设置修改。

## 18. P0-04 执行报告

报告必须包含：

1. 任务、日期、执行角色和建议状态；
2. 四个白名单文件及范围确认；
3. 实现提交完整哈希，不预测报告提交自身哈希；
4. Python 实际版本与标准库依赖确认；
5. 红灯命令、退出码、失败原因和关键输出；
6. 绿灯命令、测试方法数、退出码和关键输出；
7. 33 个行为到测试方法的映射；
8. `py_compile`、真实仓库 CLI 和两次输出一致性；
9. 各规则扫描数量和当前仓库错误数；
10. 白名单、链接、敏感信息和 `git diff --check`；
11. 异常、权限请求、风险、未决问题和计划偏离；
12. 未执行远程写操作的确认；
13. 建议状态 `submitted`，不得宣布 `accepted`。

## 19. P0-04B 独立审查

校验器会成为所有后续内容任务的公共机械门禁，属于跨模块高影响工具。P0-04 执行提交后，必须由未参与实现的独立新会话审查。

P0-04B 必须：

- 固定实现提交和报告提交完整哈希；
- 确认代码、测试、README 和报告提交范围；
- 独立运行完整单元测试、`py_compile` 和真实仓库 CLI；
- 在临时仓库独立构造正例与负例，不只复述现有测试；
- 检查 Front Matter 假阴性和假阳性；
- 检查链接越界、大小写、围栏和中文路径；
- 检查占位符排除目录是否过宽或过窄；
- 检查敏感信息消息不回显原值；
- 连续运行两次确认输出稳定；
- 阅读实现，检查异常吞噬、路径逃逸和规则重复；
- 核对执行报告与 Git 事实；
- 只创建独立审查报告，不修改实现。

问题分为阻塞、必须修改、建议修改和通过项。建议结论只允许 `passed`、`passed-with-notes` 或 `changes-required`，无权宣布 `accepted`。

## 21. 停止条件

出现任一情况立即停止并报告主会话：

- 任务状态、分支、远程或工作区不符合启动门禁；
- 目标文件已存在或同一文件正被其他会话修改；
- 需要写入白名单外文件；
- 需要修改 PRD、AI 规则、模板、版本基线或任务状态；
- 标准库无法在批准范围内正确实现某项规则；
- 测试要求与当前模板协议发生冲突；
- 真实仓库校验失败且根因位于白名单外文件；
- 敏感信息模式发现真实疑似凭据；
- 红灯不是由实现缺失导致，或绿灯无法在白名单内取得；
- 同一验证失败两次仍无法判断是实现还是测试缺陷；
- 无法形成规定提交或发现未经授权远程操作；
- 同一外部阻塞连续出现三次。

## 22. 风险与后续边界

- 单文件实现未来可能增长；只有规则显著增加并由新任务批准时才拆包；
- Front Matter 和 Markdown 采用受控子集，会主动拒绝复杂但合法的通用语法；README 必须提前说明；
- 大小写检查需要在 Windows 和 Linux 语义之间保持一致，必须由独立审查重点复验；

- P0-04 不接入 CI，未来 CI 任务只能调用同一 CLI，不复制规则；
- P0-05 可以在 P0-04 接受后继续制定，但 P0-04 不决定教学软件版本；
- P0-06 仍负责 Phase 0 总体独立审查，不被 P0-04B 替代。
