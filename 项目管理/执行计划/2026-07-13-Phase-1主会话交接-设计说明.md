# Phase 1 主会话交接设计说明

**状态：** 已批准，待书面规格复核

**设计日期：** 2026-07-13

**制定者：** Phase 0 主会话

**用户批准：** 2026-07-13

**Phase 0 最终验收提交：** `a3c04d28461861bdd166d9f2208201dae3c3011f`

## 1. 背景

Phase 0 治理与骨架已经通过最终验收。用户决定为 Phase 1 新建独立主会话，由该会话在 Phase 1 期间接替当前会话，成为唯一项目领导者、任务签发者和最终验收者。当前主会话只负责完成一次性、可追溯的正式交接；交接生效后不再参与 Phase 1 日常治理，避免两个主会话同时修改任务状态、规格或验收结论。

本设计只规定交接载体、信息边界、启动门禁和新主会话的第一步，不替 Phase 1 决定完整信息架构，也不授权批量生产学习正文。

## 2. 目标

1. 把 Phase 0 的已验收状态、关键决策、未决门禁和权限边界注入新主会话；
2. 明确 Phase 1 只有一个主会话，消除双重领导和并发治理风险；
3. 给出可直接复制的完整启动指令，使新会话不依赖旧聊天记录；
4. 要求新主会话先制定 Phase 1 设计，再制定实施计划和任务包；
5. 保留 Git、验证、版本、安全、远程操作和停止条件；
6. 为 Phase 1 结束后的下一次阶段交接建立可复用入口。

## 3. 非目标

- 不创建 Phase 1 完整章节树、学习路线、职业路线或三个月就业路线；
- 不创建 Phase 1 任务包或更新任务看板；
- 不编写学习正文、实验、项目、故障案例、面试题、脚本或图表；
- 不修改 `AGENTS.md`、`CODEX-PRD.md`、`AI-CODEX-RULES.md`、ADR 或软件版本基线；
- 不决定 VitePress 依赖版本、各阶段软件组合或 GPU/模型组合；
- 不 push、不创建 Pull Request、不发布网站、不修改 GitHub 设置或创建云资源；
- 不复制旧聊天全文作为项目记忆。

## 4. 方案比较与决定

| 方案 | 做法 | 优点 | 缺点 | 决定 |
|---|---|---|---|---|
| A | 创建专用 `项目管理/会话交接/` 目录、交接文件和管理入口 | 可追溯、可版本控制、职责清楚、后续阶段可复用 | 增加一个治理目录 | 采用 |
| B | 在仓库根目录创建单一交接文件 | 入口明显 | 根目录治理文件继续膨胀，阶段记录不易归档 | 不采用 |
| C | 只在聊天中提供启动指令 | 最快 | 新会话依赖临时上下文，无法审查和版本化 | 不采用 |

## 5. 文件结构与职责

交接实施只允许创建或修改以下三个文件：

| 文件 | 操作 | 职责 |
|---|---|---|
| `项目管理/会话交接/README.md` | 创建 | 定义交接目录用途、允许编辑者、命名规则、必要内容和安全边界 |
| `项目管理/会话交接/Phase-1-主会话交接.md` | 创建 | 保存 Phase 1 权限转移、状态快照、必读顺序、门禁、工作边界和完整启动指令 |
| `项目管理/README.md` | 修改 | 增加“会话交接”目录入口及职责说明 |

不更新任务看板，因为交接不是具体内容任务，也不代表 Phase 1 设计任务已经创建或进入 `ready`。

## 6. 权限转移模型

### 6.1 生效条件

交接在以下条件全部满足后生效：

1. 三个白名单文件按本设计完成；
2. 仓库校验、格式检查和提交范围验证通过；
3. 交接提交已经创建，完整哈希已交付给用户；
4. 用户把交接文件中的启动指令发送到新的 Phase 1 会话；
5. 新会话完成启动门禁并明确接受唯一主会话职责。

当前会话在第 1 至第 3 项完成后停止新增 Phase 1 治理决定；第 4 至第 5 项由用户和新会话完成。用户在交接生效前提出的修订仍由当前会话处理。

### 6.2 新主会话权限

新主会话在 Phase 1 期间继承 `AGENTS.md` 规定的全部主会话职责，包括：

- 维护 Phase 1 范围、路线、架构和质量门禁；
- 制定设计说明、实施计划、任务包和验收标准；
- 更新任务看板、决策记录和项目记忆；
- 审查执行会话交付物并作出最终状态裁决；
- 对高风险或阶段性成果签发独立审查任务；
- 在 Phase 1 结束时组织阶段验收和下一次主会话交接。

### 6.3 排他边界

- Phase 1 同一时刻只有一个主会话；
- 当前 Phase 0 主会话完成交接后不再签发 Phase 1 任务或更新其验收状态；
- 执行会话和独立审查会话不能因主会话更换而获得额外权限；
- 用户始终可以暂停、替换或重新授权主会话。

## 7. 交接文件内容

`Phase-1-主会话交接.md` 必须包含以下部分：

1. 交接声明与唯一主会话身份；
2. Phase 0 最终状态、最终验收报告和固定提交；
3. 新会话必读文件及顺序；
4. 启动仓库门禁和实际验证；
5. Phase 1 已授权范围；
6. 仍然禁止的范围；
7. 已接受 ADR、版本策略与项目基线摘要；
8. Kubernetes 四层版本教学策略；
9. 唯一非阻塞 GPU/模型决策门；
10. Phase 1 第一个目标与工作顺序；
11. 任务签发、独立审查和最终验收规则；
12. 停止条件、远程操作和安全边界；
13. Phase 1 结束时的交接要求；
14. 可直接复制到新会话的完整启动指令。

交接文件只保存高价值状态和路由，不复制 PRD、规则或历史报告全文。动态事实必须链接权威入口，并标注核验责任。

## 8. 新主会话必读顺序

启动指令至少要求新主会话完整读取：

1. `AGENTS.md`；
2. `项目管理/会话交接/Phase-1-主会话交接.md`；
3. `CODEX-PRD.md`，重点但不限于 Phase 1 和治理章节；
4. `项目管理/验收记录/Phase-0-最终验收报告.md`；
5. `项目记忆/MEMORY.md` 及由其路由的 `decisions.md`、`patterns.md`、`open-questions.md`、`resources.md`；
6. `项目管理/任务看板.md`；
7. `版本记录/软件版本基线.md` 与 `参考资料/来源优先级.md`；
8. ADR-0001 至 ADR-0005；
9. `项目管理/执行计划/2026-07-13-Kubernetes分层版本策略-设计说明.md`；
10. `AI-CODEX-RULES.md` 和 `模板/README.md`，因为 Phase 1 会设计后续内容任务和交付合同。

新主会话不得只依赖本交接摘要替代权威文件。

## 9. 启动门禁

新主会话必须先执行并记录：

1. 当前目录、分支、HEAD、工作区状态和远程地址；
2. Phase 0 最终验收提交与交接提交均为当前 HEAD 祖先；
3. 工作区干净，没有未知修改；
4. Python 版本与校验器基线一致；
5. 44 项仓库校验器单元测试全部通过；
6. 仓库校验器退出 0，知识章节仍为 0，错误为 0；
7. 正式内容目录仍无非 `README.md` 学习 Markdown；
8. `git ls-remote --heads origin` 只读检查按环境能力执行，禁止借此 push 或修改凭据。

任何未知修改、固定提交缺失、测试失败、校验失败或正式正文越界都必须先停止 Phase 1 设计并向用户报告。

## 10. 动态版本启动检查

新主会话启动时必须打开 Kubernetes 官方 1.36 发布线与 patch 页面，判断版本基线记录的“当前 patch”是否仍为正式最新值：

- 如果只有未来计划 patch，不修改基线；
- 如果新 patch 已正式发布，登记独立版本维护任务，不在交接流程中直接修改全局基线；
- 与 patch 无关的 Phase 1 信息架构设计可以继续，但任何“当前 patch”结论、正文、实验或任务包必须等待版本入口更新和验证；
- minor、运行时 LTS 线、CNI minor 或主实验组合变化仍必须通过替代 ADR，不能按 patch 维护处理。

同时确认 containerd 和 Calico 官方兼容入口仍覆盖 Kubernetes 1.36；交集失效时停止相关版本结论并交由 ADR 裁决。

## 11. Phase 1 授权范围

Phase 1 当前只授权设计和规划：

- 总学习导航；
- 职业成长路线；
- 三个月就业路线；
- 完整阶段与章节树；
- 技术索引；
- 知识前置依赖；
- 项目演进线与章节映射；
- 后续任务拆分、优先级、文件所有权和验收门；
- 与上述成果对应的设计说明、实施计划、任务包、审查和治理记录。

不授权批量生产正式学习正文。若 Phase 1 设计最终提出样章验证，必须先由新主会话完成设计、用户批准、实施计划和任务包，再交由独立执行会话完成。

## 12. 新主会话第一目标

新主会话的唯一首要目标是：

> 在不创建正式学习正文的前提下，审计 PRD 的 Phase 1 范围和现有目录职责，提出 Phase 1“导航与完整大纲”的设计方案，取得用户批准后形成书面设计与实施计划。

推荐顺序：

1. 复述启动门禁结果和继承边界；
2. 审计 PRD、目录 README、模板和现有路线约束；
3. 逐项确认导航、章节编号、颗粒度、依赖表达和三个月路线裁剪规则；
4. 提出两至三种信息架构方案及权衡；
5. 向用户展示推荐设计并取得批准；
6. 写入 Phase 1 设计说明，完成规格自检并提交用户复核；
7. 设计获批后制定详细实施计划；
8. 计划获批后才创建边界清晰的执行任务包。

## 13. 已知基线与未决门禁

新主会话必须保留以下事实边界：

- 文档站方向为 VitePress，具体依赖版本仍为阶段门；
- 项目采用内容 CC BY-SA 4.0、代码 Apache License 2.0 的双许可证；
- 传统项目固定为 RuoYi-Vue v3.9.2、commit `0e2d75c`，不提交完整上游源码；
- Kubernetes 概念尽量版本中立，1.36.x 为唯一主实验，1.35.x 为企业兼容参考，1.31.14 为历史升级参考；
- 阿里云预算不构成项目级阻塞，但实验必须提供资源释放步骤；
- GPU 使用云上租用资源，优先阿里云 GPU ECS；具体实例、GPU、显存、地域、价格和目标模型组合仍是 AI Infra 实验进入 `ready` 前的非阻塞决策门；
- 所有未取得环境证据的组件只能称为来源核验或待验证，不能声称已经部署成功。

## 14. 停止条件

新主会话遇到以下情况必须停止扩大范围并请求用户裁决：

- 权威文件之间存在无法按优先级消解的冲突；
- 需要改变 Phase 0 已接受的项目定位、许可证、主版本、贯穿项目或角色边界；
- 需要让两个主会话同时拥有 Phase 1 最终裁决权；
- Phase 1 设计必须依赖尚未核定的动态组件版本或 GPU/模型组合；
- 发现未知工作区修改、历史对象缺失、测试失败、校验失败或敏感信息；
- 需要 push、创建 PR、发布、采购、创建云资源或修改 GitHub 设置但没有明确授权；
- 用户要求直接批量生产正文而样章、任务包或审查门尚未建立。

## 15. Phase 1 结束与再次交接

Phase 1 新主会话在阶段结束时必须：

1. 组织独立阶段审查；
2. 对执行与审查证据作最终裁决；
3. 更新任务看板、验收记录和项目记忆；
4. 明确下一阶段真正解锁的范围；
5. 若用户继续采用分阶段主会话，创建下一阶段交接文件和可复制启动指令；
6. 在新主会话明确接任前保持唯一领导责任，不形成治理空窗。

## 16. 验收标准

1. 专用会话交接目录有职责 README 和 Phase 1 交接文件；
2. `项目管理/README.md` 有正确相对入口；
3. 权限转移生效条件和唯一主会话边界无歧义；
4. Phase 0 状态、提交、版本策略和未决门禁准确；
5. 必读顺序能把新会话路由到权威文件，而非复制长摘要；
6. 启动门禁包含 Git、测试、校验、正文空边界、动态版本和远程只读检查；
7. Phase 1 授权范围不扩大到批量正文；
8. 新主会话第一目标是设计说明，不是直接执行内容生产；
9. 完整启动指令明确角色、目录、必读文件、唯一目标、允许操作、禁止事项、验证、停止条件和首轮回复格式；
10. 没有密码、Token、Cookie、私钥、云 AccessKey 或个人敏感信息；
11. 只修改三个白名单文件；
12. 仓库校验、链接、格式和 Git 提交范围验证全部通过。

## 17. 书面规格审查门

本设计经用户书面复核后，当前主会话才制定交接实施计划。实施计划必须使用精确三文件白名单、实际验证命令、提交前后门禁和最终启动指令交付格式，不得在计划阶段直接创建 Phase 1 任务或正文。
