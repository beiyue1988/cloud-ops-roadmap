## 1. 背景

Phase 0 已完成治理、目录骨架、模板、版本与来源入口以及仓库校验工具，并只解锁 Phase 1 的学习导航、职业路线、三个月就业路线、完整章节树、技术索引、知识依赖和相关规划。正式学习正文、实验、项目实现、故障案例、面试题、脚本、配置和图表仍未授权生产。

Phase 1 启动审计确认：

- `学习路线/`、`技术索引/`、`知识库/`、`实验手册/`、`项目实战/`、`故障案例/`、`面试体系/`、`运维思维训练/` 与 `企业规范/` 的目录所有权清楚；
- 21 个知识阶段、4 级实验目录和 5 个项目目录已经建立职责 README，但尚无文件级章节清单；
- 当前缺口主要是章节编号与颗粒度、直接前置关系、三个月路线标签、职业能力映射、技术查询入口和贯穿项目里程碑映射；
- 现有目录职责不存在需要先修复的冲突，可以在不创建正式正文的前提下完成信息架构设计。

## 2. 目标与成功标准

Phase 1 采用“双重可用”作为首要成功标准：同一套章节标识既能让零基础学习者找到合理学习顺序，也能让后续主会话把章节直接拆成边界清晰的单文件任务。

具体目标：

1. 为 00 至 20 阶段建立完整的文件级知识章节清单；
2. 为每章确定稳定 ID、预定文件名、单一主要目标、直接前置、建议投入、就业标签和实践锚点；
3. 建立完整成长、三个月就业、职业成长、知识依赖和贯穿项目等学习视图；
4. 建立按技术名称、故障现象和岗位能力查询的技术索引；
5. 让三个月路线从章节就业标签和前置关系派生，不维护第二套知识正文；
6. 让实验、检查点和项目以锚点连接知识章节，不在 Phase 1 提前展开实践文件清单；
7. 建立可机械验证、可人工审查、可独立验收的质量门禁。

## 3. 非目标

- 不创建任何正式知识章节文件；
- 不创建实验、故障、面试、思维训练或企业规范的文件级内容清单；
- 不编写项目实现、命令、配置、脚本、代码、图表或实际输出；
- 不建立机器生成的导航系统，不引入 YAML/JSON 中央注册表或新的站点工具链；
- 不修改已接受 ADR、双许可证、RuoYi-Vue 固定对象或 Kubernetes 全局版本策略；
- 不选择 VitePress 具体依赖版本，不选择 GPU、模型、显存、地域或价格组合；
- 不把来源核验表述为环境验证；
- 不创建执行任务包，不更新任务状态，不执行 push、PR、发布或云资源操作。

## 4. 已比较方案

### 4.1 方案 1：阶段清单为源、导航视图派生

21 个阶段 README 各自维护本阶段文件级章节清单；学习路线、技术索引、阶段检查点和项目映射只引用章节 ID。

优点：符合现有目录职责和文件所有权；阶段可以独立理解、独立审查和独立签发；学习者入口与任务规划共享稳定标识。

代价：多个视图之间需要机械一致性检查，必须阻止重复 ID、失效引用和循环依赖。

### 4.2 方案 2：中央总清单为源

使用一个总文件保存全部章节，阶段 README 只链接中央清单。

优点：全局浏览和统计直接。

缺点：形成超大公共文件；阶段任务持续争用同一文件；局部修改也需要全局所有权，违背项目的单任务、单文件所有权模式。

### 4.3 方案 3：机器可读注册表为源

使用 YAML 或 JSON 保存章节元数据，再生成 Markdown 导航。

优点：自动化和一致性能力强。

缺点：需要提前建立生成脚本、数据 schema 和工具链；增加 Phase 1 不必要的复杂度；削弱 Markdown 的直接可读、可审查和可维护性。

### 4.4 决定

采用方案 1。Markdown 保持直接可维护；阶段 README 是章节目录的权威来源；自动化只负责验证，不负责生成内容。

## 5. 信息架构与权威来源

### 5.1 阶段章节清单

`知识库/NN-阶段名称/README.md` 是本阶段章节目录的权威来源，负责：

- 章节顺序；
- 章节 ID 与预定文件名；
- 标题、主要教学目标和知识类型；
- 直接前置章节；
- 建议投入与就业标签；
- 阶段检查点、实验等级或项目里程碑锚点。

阶段 README 不包含正式教学解释、命令、配置、实验步骤、故障答案或面试答案。

正式章节在后续 Phase 建立后，章节 Front Matter 中的 `chapter`、`title`、`type`、`prerequisites` 和 `study_time` 必须与已接受章节清单一致。阶段 README 继续负责目录顺序、就业标签和实践锚点。

### 5.2 学习路线

`学习路线/` 是面向读者的顺序视图。路线可以显示章节 ID 和短标题，但不得复制章节目标、原理或实践正文。完整章节记录仍回到对应阶段 README。

### 5.3 技术索引

`技术索引/` 是面向查询的交叉视图，只保存关键词、问题或能力与章节 ID 之间的映射，不保存教程、故障处置过程或面试答案。

### 5.4 实践与项目

- `实验手册/` 在 Phase 1 只提供 `LAB-L1` 至 `LAB-L4` 等级锚点；
- `学习路线/阶段检查点/README.md` 只定义阶段成果与检查点锚点；
- `项目实战/01` 至 `04` 形成若依传统部署到可观测性建设的演进主线；
- `项目实战/05` 保持独立 AI 推理服务项目；
- 项目 README 维护里程碑与章节 ID 的映射，但不包含实现步骤。

## 6. 章节记录合同

每个阶段 README 使用统一章节清单。每章必须包含以下字段：

| 字段 | 规则 |
|---|---|
| 章节 ID | `NN.CC`，`NN` 与阶段目录一致，`CC` 为阶段内两位序号 |
| 预定文件 | `NN.CC-章节名称.md`，名称无空格，技术名保留英文 |
| 标题 | 对零基础学习者可理解，准确限定本章范围 |
| 主要目标 | 只有一个可检查的主要教学目标 |
| `type` | 使用 PRD 已批准的知识类型集合 |
| 直接前置 | 只列必须先掌握的直接章节 ID，不重复传递性前置 |
| 建议投入 | 30–60、60–90 或 90–120 分钟 |
| 就业标签 | `必须学`、`建议学`、`就业后补学` 三选一 |
| 实践锚点 | 可为空；适用时引用检查点、实验等级或项目里程碑 ID |

颗粒度规则：

1. 一个章节对应未来一个 Markdown 文件；
2. 一个章节只承担一个主要目标；
3. 单章建议投入保持在 30–120 分钟；
4. 命令速查、独立实验、故障案例和面试题不混入知识章节清单；
5. 大主题按能够独立学习和检查的知识点拆分，不按软件名称机械合并；
6. 不为满足数量创建空章节、纯历史章节或没有学习价值的切片。

章节 ID 在 Phase 1 验收后保持稳定。确需重排或改名时，必须由独立大纲维护任务原子更新阶段清单、学习路线、技术索引、依赖和实践映射；旧 ID 不得被静默复用为无关主题。

## 7. 学习路线文件设计

### 7.1 `00-学习路线导航.md`

帮助读者根据目标选择：完整成长路线、三个月就业路线、职业专题路线或继续学习入口。只解释路线适用人群和选择方式。

### 7.2 `01-完整成长路线.md`

展示 00 至 20 的阶段顺序、阶段目标、阶段产出和阶段 README 链接。完整文件级章节树由 21 个阶段 README 共同组成，本文件不复制全部章节表。

### 7.3 `02-三个月就业路线.md`

三个月路线按以下规则派生：

1. 收集所有标记为 `必须学` 的章节；
2. 计算这些章节的全部必需前置闭包；
3. 按依赖顺序编入十二周主线；
4. `建议学` 作为弹性补充，不阻塞主线；
5. `就业后补学` 默认不进入十二周主线；
6. 若 `必须学` 章节依赖 `就业后补学` 章节，必须调整标签或收窄章节目标后才能验收。

路线只提供建议投入与顺序，不承诺固定时间掌握或就业结果。

### 7.4 `03-职业发展路线.md`

按 Linux 运维、云计算运维、云原生运维、DevOps、SRE 基础和 AI Infra 入门展示能力递进，并将岗位能力映射到章节 ID、阶段成果和项目锚点。

### 7.5 `04-每日每周学习模板.md`

提供“理解 → 操作 → 验证 → 复盘”的日节奏和“新知识 → 实验 → 综合任务 → 查漏补缺”的周节奏，不设置强制完成日。

### 7.6 `05-知识前置依赖.md`

提供阶段级主干、关键跨阶段依赖和阅读说明。章节级直接边仍以阶段 README 为权威，本文件不复制完整边集合。

### 7.7 `06-贯穿项目演进线.md`

汇总五个项目的定位、主线与独立分支、里程碑、章节 ID 和未来证据类型。项目 README 继续拥有具体里程碑记录。

### 7.8 `阶段检查点/README.md`

登记 `CP-NN` 检查点、阶段成果、所需章节 ID 和实践锚点，不创建检查题、命令或实际验收结果。

新增 `05`、`06` 和阶段检查点入口用于承载 PRD 已授权的知识依赖与贯穿项目映射，不改变 `学习路线/` 只做导航的目录职责。

## 8. 技术索引文件设计

### 8.1 `按技术名称.md`

按 Linux、Nginx、MySQL、Docker、Kubernetes、Prometheus、CUDA 等技术名组织关键词到章节 ID 的映射。别名和中英文关键词应指向同一权威章节。

### 8.2 `按故障现象.md`

按“无法连接”“服务启动失败”“磁盘空间不足”“延迟升高”等现象指向解释机制和排查方法的知识章节，同时保留未来故障案例入口，不在索引中写处置答案。

### 8.3 `按岗位能力.md`

按主机管理、网络排查、服务部署、数据维护、自动化、容器、云原生、可观测性、可靠性和 AI Infra 等能力映射章节 ID、阶段成果和项目锚点。

## 9. 知识依赖模型

章节依赖构成有向无环图：

- 依赖边从后学章节指向必须先学章节；
- 章节不得依赖自身；
- 被引用章节必须存在；
- 路线顺序必须保证前置章节先出现；
- 不允许循环依赖；
- 只记录直接前置，传递性前置由校验和路线派生计算；
- 仅有帮助但不是完成本章所必需的内容，不进入 `直接前置`，后续可作为相关内容链接。

阶段顺序仍是零基础读者的默认主干；依赖图允许阶段跳过不必要的中间阶段，例如网络基础可以直接成为 Kubernetes 的前置，但不得破坏必需知识顺序。

## 10. 实践锚点与项目映射

### 10.1 锚点格式

| 类型 | 格式 | 含义 |
|---|---|---|
| 阶段检查点 | `CP-NN` | 对应阶段 `NN` 的能力完成点 |
| 实验等级 | `LAB-L1` 至 `LAB-L4` | 预期实践深度，不指向具体实验文件 |
| 项目里程碑 | `PRJ-NN-MCC` | 项目 `NN` 的两位里程碑序号 |

### 10.2 项目关系

主线项目：

1. `PRJ-01`：若依传统部署；
2. `PRJ-02`：Docker 容器化改造；
3. `PRJ-03`：Kubernetes 云原生部署；
4. `PRJ-04`：可观测性建设。

独立高阶项目：

5. `PRJ-05`：AI 推理服务部署，不把若依业务强行改造成 AI 系统。

每个项目里程碑必须记录：

- 唯一锚点 ID；
- 可观察的能力结果；
- 所需章节 ID；
- 预期证据类型；
- 未来内容归属。

阶段章节清单引用项目锚点，项目 README 反向列出所需章节。双方必须一致，形成可追踪但不复制正文的双向映射。

## 11. 已接受基线的适用方式

### 11.1 发布与许可

- VitePress 方向保持不变，具体 VitePress、Node.js 和包管理器版本仍为文档站任务阶段门；
- 原创内容与图表继续采用 CC BY-SA 4.0；
- 原创代码与配置继续采用 Apache License 2.0；
- Phase 1 导航文件属于原创内容范围，不改变第三方材料许可证。

### 11.2 贯穿项目

RuoYi-Vue 固定为官方 Gitee tag `v3.9.2` 和 commit `0e2d75c`。章节树可以规划应用运行与演进所需知识，但不得提前选择尚未核定的 JDK、MySQL、Redis 或前端工具版本。

### 11.3 Kubernetes

章节树必须保留四层教学角色：

1. 核心概念尽量版本中立；
2. Kubernetes 1.36.x 是唯一主实验环境；
3. Kubernetes 1.35.x 是企业兼容参考；
4. Kubernetes 1.31.14 是历史升级参考。

动态 patch 不写入章节标题、章节 ID 或路线标签。1.31.6 只允许出现在明确旧案例中，不得成为推荐安装版本。主实验环境不等于企业唯一版本。

### 11.4 AI Infra

GPU 使用云上租用资源并优先阿里云 GPU ECS 的方向保持不变。实例族、GPU、显存、地域、价格、模型以及 NVIDIA/CUDA/PyTorch/vLLM 组合仍是阶段 18 实验进入 `ready` 前的非阻塞决策门，不阻塞 Phase 1 信息架构。

## 12. 自动验证设计

Phase 1 必须扩展机械校验能力，但实现只能由获得精确文件白名单的独立执行会话完成。自动门禁至少检查：

1. 21 个阶段 README 均包含合法章节清单；
2. 章节 ID 全局唯一，格式正确且与阶段目录一致；
3. 预定文件名唯一并与章节 ID、命名规则一致；
4. 标题、主要目标、`type`、建议投入和就业标签完整合法；
5. 前置 ID 存在、无自引用、无循环；
6. 路线顺序满足前置关系；
7. 三个月 `必须学` 集合包含完整前置闭包；
8. 路线、索引、检查点和项目锚点只引用有效 ID；
9. 章节与项目的双向映射一致；
10. Markdown 链接、占位符、敏感信息和现有仓库校验继续通过；
11. 新差异通过 Git 空白检查；
12. 九类内容目录没有越权产生正式教学正文。

验证工具只判断可机械证明的结构与引用关系，不替代教学、范围和版本人工审查。

## 13. 人工审查设计

人工审查至少覆盖：

- 零基础读者能否在不猜测前置概念的情况下连续学习；
- 每章是否只有一个主要目标且能够在 30–120 分钟内独立学习；
- 21 个阶段是否覆盖 PRD 范围，同时避免重复正文和空章节；
- 三个月路线是否覆盖就业主线且没有隐藏前置；
- 职业路线是否体现 Linux 运维到 AI Infra 的能力递进；
- 项目 01 至 04 是否围绕同一若依业务连续演进；
- 项目 05 是否保持独立；
- Linux 单机日志与企业集中日志是否分层；
- AI Infra 与 AIOps 是否分开；
- Kubernetes 四层版本角色是否准确；
- 技术索引能否从技术名、故障现象和岗位能力找到权威章节；
- 未核定动态版本、GPU 或模型是否仍保持阶段门状态。

阶段性成果必须由独立审查会话检查固定提交；审查建议不能替代主会话最终裁决。

## 14. 错误处理与停止条件

以下问题阻止相关任务或 Phase 1 进入验收：

- 重复、缺失、格式错误或被静默复用的章节 ID；
- 不存在的前置、路线、索引或项目引用；
- 自引用或循环依赖；
- `必须学` 章节的前置闭包包含未处理的 `就业后补学` 章节；
- 单章承担多个不可独立验收的主要目标；
- 路线或索引复制正式知识正文；
- 创建了未授权实验、故障、面试、配置、脚本、图表或项目实现；
- 权威规格、ADR、版本基线或目录职责冲突且无法消解；
- Phase 1 设计必须依赖未核定动态版本、GPU 或模型才能成立；
- 需要修改已接受 ADR、许可证或全局版本策略；
- 测试、仓库校验、链接、安全扫描或独立审查存在未关闭阻塞项；
- 需要外部写操作但用户未明确授权。

### 15.5 波次 5：独立阶段审查

### 15.6 波次 6：主会话最终验收

主会话独立复验执行与审查证据后，才可更新任务看板、验收记录和项目记忆，并决定 Phase 2 真正解锁的范围。
