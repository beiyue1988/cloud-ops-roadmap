# Phase 1 波次 4 压缩执行与实践映射合同设计说明

## 1. 决策摘要

波次 4 延续波次 3 的压缩治理模式，将原 `P1-04A`、`P1-04B`、`P1-04C`、`P1-04-01` 至 `P1-04-05`、`P1-04D` 共十个产品任务压缩为两个交付包：

1. `P1-04-ANC`：四级实验锚点与 21 个阶段检查点；
2. `P1-04-PRJ`：五项目入口、20 个项目里程碑、贯穿项目视图、阶段章节反向锚点和路线/岗位索引项目锚点；
3. `P1-04R`：只做一次 `complete` 集成独立审查；
4. 只有 Critical/Important 才允许唯一合并维护包 `P1-04-M1`。

原计划遗漏了一个会导致必然返工的机械约束：`complete` 门禁要求项目里程碑列出的章节反向引用同一 `PRJ-*`，但波次 2 结束时 620 章中的 `PRJ-*` 为 0，原波次 4 文件表又未授权更新阶段 README。本设计明确把“只修改实践锚点单元格”的阶段 README 更新纳入 `P1-04-PRJ` 原子交付。

## 2. 不变边界

- 不创建实验正文、项目实现、命令、配置、代码、图表、故障演练或云资源；
- 不改变章节 ID、标题、主要目标、type、直接前置、投入或就业标签；
- 阶段 README 只允许在既有九列表的“实践锚点”单元格追加 `PRJ-*`；
- 不修改许可证、ADR、RuoYi-Vue 固定对象或 Kubernetes 四层策略；
- 不选择 JDK、MySQL、Redis、前端工具、GPU、显存、地域、价格、模型或 NVIDIA/CUDA/PyTorch/vLLM 组合；
- 自动化只验证映射，不生成内容或注册表。

固定输入为波次 3 验收提交 `be9e910aef8730609bbff3b6345ad85633797b4d` 可达的 21/620/817 章节树和 views 视图。

## 4. 实验等级合同

### 4.1 文件

```text
实验手册/README.md
实验手册/Level-1-基础实验/README.md
实验手册/Level-2-综合实验/README.md
实验手册/Level-3-企业实验/README.md
实验手册/Level-4-架构实验/README.md
```

### 4.2 四级定义

| 锚点 | 深度 | 能力边界 | 允许的证据类型 |
|---|---|---|---|
| `LAB-L1` | 单知识点观察与验证 | 单对象、低风险、可快速恢复 | 操作记录、状态输出、前后对比 |
| `LAB-L2` | 多知识点组合 | 多组件但单主机或受控范围 | 部署验证、依赖验证、回滚记录 |
| `LAB-L3` | 企业运行与故障处理 | 多节点、运行维护、受控故障和恢复 | 运行证据、故障定位、恢复验证 |
| `LAB-L4` | 架构设计与取舍 | 容量、可靠性、安全、成本和方案比较 | 设计记录、权衡说明、演练方案 |

每个 Level README 只能出现对应锚点字面量一次，不得出现其他等级锚点；只定义准入、退出、风险和证据，不列实验文件。

## 5. 阶段检查点合同

`学习路线/阶段检查点/README.md` 使用精确表头：`检查点`、`阶段`、`所需章节`、`实践锚点`、`能力结果`。

- 精确 21 行，按 `CP-00` 至 `CP-20` 连续排序；
- 阶段列精确为 `00` 至 `20`；
- 所需章节精确使用该阶段最后一章，即阶段全量闭环章；
- 实践锚点等于该阶段九列表中实际出现的 LAB 等级去重集合；没有则写 `—`；
- 能力结果从阶段产出压缩为单行可观察结果，不复制章节目标；
- 检查点不引用尚未定义的 `PRJ-*`，项目证据通过项目映射独立维护。

## 6. 项目里程碑合同

### 6.1 固定项目关系

- 项目 01–04 围绕同一 RuoYi-Vue 业务连续演进；
- 项目 01 固定上游为 RuoYi-Vue `v3.9.2` / commit `0e2d75c`；
- 项目 02 继承项目 01 的业务，不重复传统部署；
- 项目 03 继承项目 02 的镜像与运行边界，Kubernetes 保持四层版本角色；
- 项目 04 继承项目 03 的运行对象，增加日志、指标、追踪、告警和可靠性证据；
- 项目 05 是独立 AI 推理服务，不把若依改造成 AI 系统，不依赖具体 GPU/模型组合。

### 6.2 固定里程碑数量与主题

每个项目精确四个里程碑，共 20 个：

| 项目 | M01 | M02 | M03 | M04 |
|---|---|---|---|---|
| 01 若依传统部署 | 环境与运行边界 | 应用与数据服务 | 发布、备份与回滚 | 运行验证与交接 |
| 02 Docker 容器化改造 | 容器化评估与镜像 | Compose 网络、存储与配置 | 容器化运行迁移 | 验证、回退与运维交接 |
| 03 Kubernetes 云原生部署 | 集群与应用边界 | 配置、密钥、存储与网络 | 工作负载、服务与发布 | 运维、升级与回退证据 |
| 04 可观测性建设 | 信号与身份规划 | 指标、告警与仪表盘 | 日志、追踪与跨信号关联 | SLI/SLO 接口与运营闭环 |
| 05 AI 推理服务部署 | 兼容链、资源与成本门 | 推理服务与容器边界 | GPU 调度与服务观测 | 容量、安全、退出与复盘 |

项目 README 使用精确表头：`里程碑`、`能力结果`、`所需章节`、`证据类型`、`未来内容归属`。里程碑必须按 `PRJ-NN-M01` 至 `M04` 连续，每行引用 2–8 个有效章节。

证据类型只使用：设计记录、部署验证、运行证据、回滚证据、观测证据、复盘记录。未来内容归属必须指向当前项目目录，不得创建实现文件。

### 6.3 章节选择与继承

- 项目 01 主要消费阶段 09–12 的企业服务、数据、协作和云资源章节；
- 项目 02 主要消费阶段 13，并通过项目继承复用项目 01，不重复所有基础章节；
- 项目 03 主要消费阶段 14–15，并通过项目继承复用项目 02；
- 项目 04 主要消费阶段 16、17、20，并通过项目继承复用项目 03；
- 项目 05 主要消费阶段 17、18、20，保持独立；
- 同一章节在同一项目内最多进入一个里程碑；跨项目复用只允许共享的观测、可靠性或安全能力，并必须显式双向映射。

## 7. 原子双向映射合同

`P1-04-PRJ` 必须在一个产品提交中同时完成：

1. 五个项目 README 的 20 个里程碑定义；
2. `学习路线/06-贯穿项目演进线.md` 的 20 行镜像，项目、里程碑、章节集合和证据类型与项目 README 完全一致；
3. 所有被项目里程碑引用章节的阶段 README 实践锚点追加同一 `PRJ-*`；
4. 未被项目引用章节不得新增 `PRJ-*`；
5. 阶段章节和项目里程碑双向集合完全相等；
6. `学习路线/03-职业发展路线.md` 与 `技术索引/按岗位能力.md` 的项目锚点列同步补充已定义里程碑。

实践锚点单元格顺序固定为 `CP-*`、`LAB-*`、`PRJ-*`。不得改变该行其他八个单元格。

职业/岗位映射固定为：

- 运维认知与主机基础：项目 01 M01；
- Linux 系统与自动化运维：项目 01 M02–M04；
- 企业服务、云平台与容器：项目 01 M04、项目 02 M01–M04；
- Kubernetes 与 DevOps：项目 03 M01–M04；
- 可观测性与 SRE：项目 04 M01–M04；
- AI Infra：项目 05 M01–M04；
- AIOps/MLOps 继续成长：`—`，不得用项目 05 冒充 AIOps 项目。

## 8. 贯穿项目视图与入口

`学习路线/06-贯穿项目演进线.md` 使用表头：`项目`、`里程碑`、`所需章节`、`证据类型`，精确 20 行。

`项目实战/README.md` 一次性说明：五项目入口、01–04 连续演进、05 独立、里程碑格式、证据与未来内容归属、无实现边界。所有链接必须有效。

- ANC：133/133、默认、views、partial、21 个 CP、4 个 LAB、正式正文和 Git 范围；

## 10. 返工上限

Minor 且不影响 complete、双向集合或用户可用性的事项不阻塞。所有 Critical/Important 只进入一个 `P1-04-M1`；M1 后只定向复核发现闭合。同一阻塞仍未关闭则停止裁决，不创建滚动 R2/R3。

## 11. 授权依据

用户于 2026-07-16 要求继续推进，并已明确要求后续波次一次性固定约束、减少重复审查和返工。本设计依该治理方向压缩波次 4，不扩大全局范围。
