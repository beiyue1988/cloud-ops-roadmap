## 1. 计划权限与执行规则

本文件是主会话的实施编排，不是可由单个执行会话直接领取的任务包。只有在用户批准本计划后，主会话才能按波次创建任务包，并把对应任务加入 `项目管理/任务看板.md`。

首要成功标准仍是“双重可用”：学习者可以按清晰前置连续阅读，主会话也可以按稳定章节 ID 直接签发单文件正文任务。章节直接前置共同构成有向无环图，任何派生路线都必须保持该依赖顺序。

执行期间必须遵守：

1. 一个任务包只对应一个主要成果和一次独立验收；
2. 每个执行会话只修改任务包逐项列出的文件及自己的执行报告；
3. 同一文件只允许一个活动写会话，公共导航与治理文件默认串行；
4. 每个任务必须从已接受提交开始，并提交独立、可审查的本地 Git 提交；
5. 执行会话只建议 `submitted`、`rework` 或 `blocked`，主会话才可写 `accepted`；
6. 高风险校验器、阶段组集成、跨视图映射和 Phase 1 总成果必须由独立审查会话检查固定提交；
7. 用户未授权 push、Pull Request、标签、Release、网站发布、GitHub 设置或云资源写操作；
8. 任何任务不得创建知识章节正文、实验步骤、项目实现、故障案例、面试答案、脚本示例、配置、命令教程或图表；
9. GPU、模型、显存、实例、地域和价格不进入本计划的成立条件；
10. Kubernetes 动态 patch 不进入章节 ID、标题或路线标签。

## 3. 可机械解析的 Markdown 合同

### 3.1 阶段章节清单

每个阶段 README 必须保留现有阶段目标、主要主题、前置阶段、阶段产出、导航和边界，并增加唯一的 `## 章节清单` 二级标题。标题下第一张表必须使用以下精确表头：

```markdown
| 章节 ID | 预定文件 | 标题 | 主要目标 | type | 直接前置 | 建议投入 | 就业标签 | 实践锚点 |
|---|---|---|---|---|---|---|---|---|
| `04.01` | `04.01-Linux是什么.md` | Linux 是什么 | 解释 Linux 在服务器运维中的定位 | `concept` | `03.04` | 30–60 分钟 | 必须学 | `CP-04`, `LAB-L1` |
```

表格语法固定为：

- 章节 ID 使用反引号包裹的 `NN.CC`；`NN` 必须等于目录阶段号，`CC` 从 `01` 连续递增；
- 预定文件使用反引号包裹的 `NN.CC-章节名称.md`，不得含空格，且全仓库唯一；
- `type` 从 `concept`、`principle`、`service`、`platform`、`tool`、`language`、`cloud-service`、`hardware`、`methodology` 中选择；
- 直接前置为空时写 `—`，否则按学习顺序写逗号加空格分隔的反引号章节 ID；
- 建议投入只能是 `30–60 分钟`、`60–90 分钟` 或 `90–120 分钟`；
- 就业标签只能是 `必须学`、`建议学` 或 `就业后补学`；
- 实践锚点为空时写 `—`，否则使用逗号加空格分隔的反引号锚点；
- 标题和主要目标必须为单行非空文本，不得包含表格竖线、HTML 换行或正式教学解释；
- 直接前置只能指向已定义且排序在当前章节之前的章节，不允许自引用、前向依赖或循环。

### 3.3 三个月路线派生规则

机械门禁按以下算法计算十二周主线：

1. 取全部 `必须学` 章节；
2. 递归加入它们的直接前置，得到完整前置闭包；
3. 主线章节集合必须与该闭包完全相等，且每个 ID 只出现一次；
4. 按周次和单元格内顺序检查拓扑顺序；
5. 弹性章节只能来自 `建议学`，不得与主线重复；
6. `就业后补学` 不得进入主线；
7. 任一 `必须学` 章节依赖 `就业后补学` 时，章节树本身即失败，不允许路线绕过。

路线必须恰好包含第 1 周至第 12 周，不承诺固定时间掌握或就业结果。

### 3.4 实践锚点闭合规则

- `CP-NN` 只能由 `学习路线/阶段检查点/README.md` 定义，`NN` 必须是 `00` 至 `20`；
- `LAB-L1` 至 `LAB-L4` 分别由对应 Level README 定义；
- `PRJ-NN-MCC` 只能由项目 `NN` 的 README 定义，`NN` 为 `01` 至 `05`，`CC` 从 `01` 连续递增；
- 章节清单引用的每个锚点必须存在；
- 项目里程碑列出的每个章节必须在对应章节行反向引用该里程碑；
- 章节行引用的每个项目里程碑也必须在项目 README 中列回该章节；
- 项目 01–04 构成若依演进主线，项目 05 保持独立，不允许建立伪造的业务继承关系。

## 4. 校验器分级门禁

现有默认命令保持向后兼容：

```powershell
python -B '脚本资源/项目校验/validate_repository.py' .
```

新增参数 `--outline-gate`，取值和责任如下：

| 取值 | 使用时机 | 必须验证 |
|---|---|---|
| `partial` | 单个阶段或单个视图任务 | 已存在表格的结构、字段、排序、重复、可解析引用；不要求尚未进入波次的文件存在 |
| `catalogs` | 21 个阶段清单完成后 | 21 个阶段全部存在合法清单、全局 ID 和文件名唯一、依赖存在且无环、标签闭包合法 |
| `views` | 学习路线与技术索引完成后 | `catalogs` 全部规则，加上规定路线和索引文件存在、引用有效、十二周派生闭合 |
| `complete` | 实践与项目映射完成后 | `views` 全部规则，加上检查点、实验等级、项目里程碑存在且双向映射一致 |

新增诊断使用 `OL001` 至 `OL012`：

| 规则 | 含义 |
|---|---|
| `OL001` | 当前门禁缺少规定文件、标题或表格 |
| `OL002` | 表头、列数、单元格分隔或受控语法错误 |
| `OL003` | 章节 ID 格式、阶段号、连续性或全局唯一性错误 |
| `OL004` | 预定文件格式、ID 对齐或全局唯一性错误 |
| `OL005` | 标题、主要目标、`type`、投入或就业标签非法 |
| `OL006` | 直接前置缺失、自引用或前向引用 |
| `OL007` | 章节依赖存在循环 |
| `OL008` | 路线或索引引用不存在，或路线顺序违反前置关系 |
| `OL009` | 三个月主线集合、前置闭包或就业标签冲突 |
| `OL010` | 锚点格式、定义位置、连续性或引用不存在 |
| `OL011` | 章节与项目里程碑双向映射不一致 |
| `OL012` | 同一派生视图中出现禁止的重复标识或越界集合成员 |

执行 P1-01 时至少增加 28 项单元测试，保留现有 44 项，最终测试总数不得低于 72，且全部通过。新增测试必须覆盖每个 `OL` 规则、四级门禁各一个通过样例与一个失败样例、十二周集合闭包、循环依赖、ID/文件重复、锚点闭合、项目双向映射和确定性输出。

## 9. 提交与状态规则

每个执行任务使用两个本地提交：

1. 产品提交：只含任务负责的正式目录 README、导航文件、索引文件或校验器文件；
2. 报告提交：只含该任务执行报告，并在报告正文记录产品提交完整哈希。

独立审查使用单独报告提交。主会话在复验后再用治理提交更新任务看板和验收记录。禁止 amend、rebase、reset 或重写已经被任务包、执行报告或审查报告引用的提交。

任务状态转换遵守：

```text
planned → ready → in_progress → submitted → accepted
                             ↘ rework
                             ↘ blocked
```

主会话只在证据闭合后写 `accepted`。计划复选框只在对应任务最终接受后由主会话更新。

## 10. 动态版本与来源门禁

- Kubernetes 章节和项目任务启动时重新核验官方 1.36 发布线、patch 页面、containerd 与 Calico 入口；
- 新的 1.36 patch 只触发独立版本维护任务，不在章节树或项目映射任务中直接修改全局基线；
- minor 或 Kubernetes/containerd/Calico 主组合变化必须走替代 ADR；
- RuoYi 项目映射固定引用官方 Gitee tag `v3.9.2` 和 commit `0e2d75c`；
- AI Infra 只规划版本中立知识与能力依赖，不选择 GPU、模型、显存、实例、地域、价格或软件组合；
- VitePress、Node.js 与包管理器具体版本仍是文档站任务门，不属于 Phase 1；
- 来源以官方文档、标准、RFC、上游仓库和厂商一手资料为主，博客、聚合文章和 AI 输出不得作为唯一依据。
