# Phase 1 波次 3 压缩执行与派生合同设计说明

## 1. 决策摘要

Phase 1 波次 3 不再把 `P1-03A` 至 `P1-03H` 作为八个独立签发、八次独立验收的执行链。经用户于 2026-07-16 批准，执行颗粒度压缩为：

1. `P1-03-RTE`：一次性交付学习路线入口与 `00` 至 `05` 六个路线文件，覆盖原 `P1-03A` 至 `P1-03E`；
2. `P1-03-IDX`：一次性交付技术索引入口与三类索引，覆盖原 `P1-03F` 至 `P1-03H`；
3. `P1-03R`：对两个固定交付包做一次集成独立审查；
4. 只有存在 Critical/Important 时才签发一个合并维护包 `P1-03-M1`，不按文件或发现逐项拆返工。

本设计只调整波次 3 的任务颗粒度、顺序和审查次数，不改变已批准的信息架构、文件职责、表头、Phase 1 范围、许可证、版本策略或后续波次成果。

## 2. 目标与非目标

### 2.1 目标

- 在生产视图前一次性冻结派生算法、集合规模、表头和维护边界；
- 让共享入口与全部目标文件在同一个产品提交中形成，避免临时失效链接；
- 让学习路线和技术索引各自只有一个文件所有者；
- 让执行报告只提供差异和验证证据，不重复粘贴 620 章数据；
- 把高风险人工判断集中到一次波次集成审查；
- 最多进行一次合并返工，避免重复审查同一稳定内容。

### 2.2 非目标

- 不修改 21 个阶段 README、620 章、817 边或就业标签；
- 不创建正式知识正文、实验、项目实现、故障案例或面试答案；
- 不引入 YAML/JSON 注册表、生成器或新站点依赖；
- 不定义波次 4 才拥有的 `PRJ-*` 里程碑；
- 不选择 GPU、模型、云资源或动态软件组合；
- 不修改已接受 ADR、许可证、RuoYi-Vue 固定对象或 Kubernetes 全局版本策略。

## 4. 通用派生合同

1. 阶段 README 是章节目录权威来源；路线和索引是派生视图。
2. 视图只复制章节 ID 和必要短标题，不复制主要目标、原理、命令、配置、实验步骤或答案。
3. 所有章节 ID、实践锚点使用反引号；同一单元格多个标识用逗号加空格；空集合写 `—`。
4. 视图中的章节必须存在于固定 620 章集合；不得发明、重命名或静默替换 ID。
5. 波次 3 不创建或引用尚未定义的 `PRJ-*`；项目锚点列统一写 `—`。
6. 共享入口只在所属交付包中修改；另一个交付包不得回写。
7. 自动化只验证结构、引用、集合与映射，不生成 Markdown 或中央数据源。
8. 执行会话自己创建的白名单文件属于本任务交付，不得在完成检查时误判为未知外部修改；任何白名单外修改仍立即阻断。

## 5. 学习路线套件合同

### 5.1 文件集合

```text
学习路线/README.md
学习路线/00-学习路线导航.md
学习路线/01-完整成长路线.md
学习路线/02-三个月就业路线.md
学习路线/03-职业发展路线.md
学习路线/04-每日每周学习模板.md
学习路线/05-知识前置依赖.md
```

七个文件在同一个产品提交中形成。`README.md` 和 `00-学习路线导航.md` 必须链接当前提交创建的 `01` 至 `05`，不得留下计划链接、失效链接或等待后续补链接的入口。

### 5.2 完整成长路线

`01-完整成长路线.md` 精确包含 00 至 20 共 21 个有序阶段记录。每条只提供阶段、能力定位、阶段产出和阶段 README 链接；完整章节表仍回到阶段 README。

### 5.3 十二周就业路线

必须满足：

- 主线集合精确等于 293 章必须学递归闭包，每个 ID 只出现一次；
- `02.18`、`07.12` 位于主线，不同时位于弹性集合；

### 5.4 职业成长路线

固定表头为：`能力层级`、`岗位方向`、`章节 ID`、`阶段成果`、`项目锚点`。

2. Linux 系统与自动化运维；
3. 企业服务、云平台与容器；
4. Kubernetes 与 DevOps；
5. 可观测性与 SRE；
6. AI Infra；
7. AIOps/MLOps 继续成长。

00 至 20 每个阶段至少通过一个有效章节 ID 进入某个层级；AI Infra 与 AIOps/MLOps 明确为就业后成长方向，不反向进入十二周主线。项目锚点统一写 `—`，由波次 4 维护。

### 5.5 每日每周学习模板

只提供“理解 → 操作 → 验证 → 复盘”的日节奏和“新知识 → 实验 → 综合任务 → 查漏补缺”的周节奏，并说明可按个人时间缩放；不创建实验步骤，不设置强制完成日。

### 5.6 知识前置依赖

固定表头为：`后学章节`、`直接前置`、`关系说明`。

必须完整展示全部 52 条跨阶段直接边，按后学章节合并为一行；每个后学章节只出现一次。只列该章节产品表中真实存在的跨阶段直接前置，不复制 765 条阶段内边，不增加传递性边。关系说明只解释学习依赖必要性，不写知识正文。

## 6. 技术索引套件合同

### 6.1 文件集合

```text
技术索引/README.md
技术索引/按技术名称.md
技术索引/按故障现象.md
技术索引/按岗位能力.md
```

四个文件在同一个产品提交中形成，入口一次性链接三类索引。

### 6.2 技术名称索引

固定表头为：`技术主词`、`别名`、`章节 ID`。

- `tool`、`service`、`platform`、`language`、`cloud-service`、`hardware` 六类共 146 个章节必须全部至少被一个技术主词覆盖；
- 同一技术主词只出现一行，别名不得为空；中英文、缩写和常用旧称归到同一权威主词；
- 阶段清单存在相应主题时，必须覆盖 Linux、systemd、journald、rsyslog、SSH、DNS、HTTP/HTTPS、TLS、Shell/Bash、Python、Git、Ansible、Nginx、MySQL、Redis、Kafka、阿里云、Docker、Compose、containerd、Kubernetes、CI/CD、Elastic/Elasticsearch、OpenSearch、Loki、Prometheus、Alertmanager、Grafana、OpenTelemetry、CUDA、NVIDIA GPU、PyTorch、vLLM、RAG、MLOps、AIOps；
- 可以引用额外有效章节，但不得把机制说明或操作步骤写入索引。

### 6.3 故障现象索引

固定表头为：`故障现象`、`机制章节`、`排查方法章节`、`未来案例入口`。

至少覆盖主机启动与资源、网络与 DNS、Linux 服务与日志、性能与安全、企业基础服务、数据服务、Docker、Kubernetes、CI/CD、日志与可观测性十个问题域。每行必须有有效机制章节和排查方法章节；只描述可观察现象，不写处置过程。未来案例入口统一写 `—`。

### 6.4 岗位能力索引

固定表头为：`岗位能力`、`章节 ID`、`阶段成果`、`项目锚点`。

岗位能力必须覆盖学习路线套件冻结的七个能力层级，名称唯一，章节均有效，阶段成果为短描述。项目锚点统一写 `—`，由波次 4 原子补充。不得把未决 GPU/模型/云资源组合设为岗位能力成立条件。

## 7. 执行、验证与审查

### 7.1 `P1-03-RTE`

使用固定源独立派生路线。运行 133/133、默认校验、`catalogs`、`partial`、正式正文边界和集合复算。产品与报告分为两个提交。主会话只做机械初验并固定其提交，不单独签发审查。

### 7.2 `P1-03-IDX`

在 RTE 固定提交之后执行，消费职业层级但不修改学习路线。运行 133/133、默认、`catalogs`、`views`、技术类型覆盖 146/146、十类故障域与七类岗位能力检查。产品与报告分为两个提交。

### 7.3 `P1-03R`

只审查固定的 RTE 与 IDX 完整提交，唯一写入为集成审查报告。必须独立运行 133/133、默认、`views`、正式正文和 Git 范围，并人工复核零基础连续性、十二周范围、职业递进、索引可发现性和正文复制边界。

## 8. 返工上限与裁决规则

- Minor 且不影响可用性、门禁或范围的事项记录到后续维护，不阻塞波次 3；
- 所有 Critical/Important 合并进入唯一 `P1-03-M1`，不按文件或发现拆任务；
- M1 后只定向复核发现是否关闭，不重审已通过的稳定集合；
- 同一阻塞在合并返工后仍未关闭，停止并交由主会话或用户裁决，不继续创建 R2、R3 式滚动返工；
- 任何源章节树缺陷必须另行签发大纲维护任务，不得混入视图返工。

## 9. 用户批准

用户于 2026-07-16 批准本压缩方案：先一次性冻结相关约束，再以两个交付包完成波次 3，只做一次集成审查，避免重复返工和无价值 Token 消耗。
