## 2. 收尾裁决原则

1. P0-06 的 `recommend-accept` 是独立建议，主会话必须重新验证后才能接受；
2. P0-05 和 P0-06 可以在同一最终验收报告中裁决，但各自提交、范围和证据必须分开；
3. Kubernetes 分层策略发生在 P0-06 固定截止之后，报告必须把它标为“用户批准的后续解释”，不能冒充 P0-06 已审对象；
4. 分层策略不改变 Kubernetes 1.36 主实验 minor、containerd 2.3、Calico 3.32 或 1.31 `reference-only` 状态，因此不触发 P0-05 返工；
5. 如果执行日官方已经发布新的 Kubernetes 1.36 patch，而版本基线仍把旧 patch 写成“当前”，必须停止最终接受并签发 patch 更新任务；
6. P0-06 的 S-01 作为非阻塞建议采纳到 `项目记忆/patterns.md`，不改写历史提交；
7. Phase 0 接受只解锁 Phase 1 的设计、导航、章节树和任务包，不授权批量内容生产；
8. 本计划不执行 push、Pull Request、发布或 GitHub 设置修改。
