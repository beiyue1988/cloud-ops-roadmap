## 2. 目标与非目标

### 2.1 唯一目标

在不改变四级大纲门禁、`OL001`–`OL012` 责任、公开 API、默认 CLI 或正式内容结构的前提下，把“行级直接诊断”“记录级闭合资格”“表级完整性”和“确定性 SCC 输出”分离，使一条不合格记录只隔离自身或能够明确界定的最小域，不再吞掉无关合法记录的诊断。

### 2.2 非目标

- 不新增或修改正式学习大纲、路线、索引、检查点或项目内容；
- 不引入完整 Markdown、HTML 或 YAML 解析器；
- 不改变表头、字段、锚点格式、门禁名称或诊断编号；
- 不拆分新模块，不引入第三方依赖、schema、CI 或配置；
- 不修改 Phase 0 已接受决定、许可证、版本策略或动态版本事实；
- 不通过弱化 README 的 `O(V+E)` 承诺回避 SCC 实现问题。

## 3. 采用方案

采用“有界内部资格模型”，而不是继续叠加全局布尔条件，也不进行跨模块重构。

### 3.1 直接诊断与状态登记分离

受控表解析继续返回表结构状态和所有结构可解析的数据行。无论整表是否具备集合比较资格，每个结构可解析行都必须执行与本行字段有关的直接验证；表结构错误只禁止该表参与表级集合、顺序和闭合推导，不能阻止其他行报告自身的 `OL002`、`OL003`、`OL004`、`OL005`、`OL008`、`OL010` 或 `OL012`。

行级验证先形成内部候选记录，候选至少表达：来源路径和行号、规范记录键、已解析值，以及结构、主键和章节集合三类资格。只有满足对应闭合前提的候选才登记到章节、锚点或项目闭合状态；直接诊断不依赖候选能否登记。

结构失败后的可解析行只执行字段格式、非空和本行键域等直接验证，不得因为前一坏行被跳过而重新编号并派生连续性、覆盖、重复、顺序或集合错误。表级位置和顺序检查只在整表结构合格时运行；实现如需保留数据区物理序号，必须将它与“有效行列表中的下标”分开。

所有现有 `structure_valid` 调用点都要审计。实现可以保留 `_ControlledTable`，但不能再用同一个条件同时控制“是否验证行”和“是否参与派生比较”。

### 3.2 最小失资格域

项目总视图和项目 README 使用记录级资格：

- 章节字段非法时，只隔离该里程碑记录；其他已完整解析里程碑继续参与 D=V、逐里程碑章节集合和章节双向映射；
- 项目 README 整表结构非法时，只使该项目侧失去闭合资格，其他项目继续比较；该表中结构可解析行仍执行直接字段验证；
- 项目总视图整表结构非法时，该视图不执行 D=V 集合推导，但结构可解析行仍执行直接字段验证；
- 非法里程碑不得登记为有效锚点定义或项目映射；失资格身份必须同时包含来源侧、记录所有者和该行的原始/预期锚点，不能退化为全局字符串集合，不能通过读取错误锚点内嵌项目号把资格转移到另一个项目；如果另一个项目合法定义了与错误原始键同名的里程碑，该合法记录仍必须参与闭合；
- 同一项目中一条非法 M02 不得使合法 M01 失去闭合资格；一个项目的不合格记录不得吞掉另一个项目真实缺失的 `OL011`。

当项目总视图里程碑字段非法时，内部候选必须保留足以隔离该行的稳定身份。对于现有 `项目 NN` 与 `PRJ-xx-Mcc` 组合，可用项目列、里程碑后缀和行上下文得到预期键；原始错误键和预期键都只作为该行的隔离键。无法得到稳定身份时，允许停止该表的 D=V 集合推导，但仍不得停止其他行的直接验证。

### 3.3 检查点和其他受控表

阶段检查点表结构失败时，`CP-00`–`CP-20` 的闭合推导不得把该结构错误放大为引用不存在的 `OL010`；结构可解析行仍报告自身直接错误。阶段清单、路线、索引和其他受控表也遵循相同的两阶段原则：先验证结构可解析行的直接字段，再根据表级资格决定是否登记或执行集合、顺序和闭合推导。

### 3.4 SCC 线性后处理

确定性节点顺序只在邻接归一化阶段建立一次，并生成节点排名。两遍非递归 SCC 遍历继续保持 `O(V+E)`。遍历后不得再调用 `sorted(members)`、`sorted(components)` 或等价的比较排序。

确定性输出采用线性装桶：遍历得到 `node -> component` 后，按已归一化节点顺序扫描一次，将节点追加到对应分量；第一次遇到循环分量时确定分量输出顺序。这样成员顺序、分量顺序和最小章节 ID 定位都来自同一规范排名，后处理时间与附加内存均为 `O(V+E)`。

## 4. 诊断数据流

```text
Markdown 屏蔽与受控表扫描
        ↓
表结构结果 + 可解析数据行
        ↓
逐行直接验证 ─────────→ 直接 OL 诊断
        ↓
形成带资格与隔离键的候选记录
        ↓
按最小失资格域登记有效状态
        ↓
D=V / 双向映射 / 锚点 / 集合与顺序推导
        ↓
只针对合格且无关的记录产生派生诊断
```

## 5. 冻结行为矩阵

| 场景 | 必须保留 | 必须禁止 |
|---|---|---|
| 一行表边界非法，另一行字段非法 | 两行各自直接诊断 | 因整表跳过而漏报第二行 |
| 检查点表结构非法且章节引用 CP | 表结构直接 `OL002` | 派生“CP 不存在”`OL010` |
| PRJ-01 视图章节非法，PRJ-05 合法映射缺少视图行 | PRJ-01 直接 `OL002` + PRJ-05 `OL011` | 全局关闭 D=V |
| PRJ-01 项目章节非法，PRJ-05 合法映射缺少视图行 | PRJ-01 直接 `OL002` + PRJ-05 `OL011` | 因任一 invalid 集合关闭全部比较 |
| 项目 01 的第二行使用错误 `PRJ-02-M02` | 该行直接 `OL010` | 登记错误锚点或派生额外 `OL011` |
| 项目 01 错用 `PRJ-02-M02`，项目 02 同时合法定义该键 | 项目 02 合法记录及其真实闭合诊断 | 以全局失资格键吞掉项目 02 |
| 项目 01 的 M02 非法，合法 M01 仍存在映射缺口 | M02 `OL010` + M01 `OL011` | 让整个项目 01 失资格 |
| 完整合法表干净删除一行 | 对真实缺项产生 `OL011` | 以资格隔离掩盖真实缺项 |
| 稀疏深图、稠密 DAG、密集 SCC | 确定、无递归、每 SCC 一条诊断 | 邻接归一化后比较排序或路径物化 |

## 6. 测试设计

R6 实施计划固定新增 11 个独立测试方法，当前 R5 产品应全部因对应缺口红灯，原 118 项保持通过；修复后总数不得低于 129：

1. 阶段表结构失败仍保留另一可解析行的直接字段诊断；
2. 波次三视图表结构失败仍保留另一可解析行的直接字段诊断；
3. 项目总视图结构失败仍保留另一可解析行的直接诊断；
4. 项目 README 结构失败仍保留另一可解析行的直接诊断；
5. 检查点结构失败不派生 CP 引用不存在诊断，并保留可解析行的直接诊断；
6. 非法项目视图章节记录不隐藏无关项目真实缺失的 `OL011`；
7. 非法项目 README 章节记录不隐藏无关项目真实缺失的 `OL011`；
8. 非法项目视图里程碑不隐藏无关项目真实缺失的 `OL011`；
9. 错误项目号里程碑只产生本行直接错误；同一方法的第二子例同时建立另一个项目的合法同名里程碑及真实映射缺口，证明错误记录既不登记定义，也不让全局失资格键吞掉合法记录；
10. 非法兄弟里程碑不关闭同项目合法里程碑的 `OL011`；
11. SCC 结构证据证明归一化后没有成员或分量比较排序，并保留深链、稠密 DAG、密集 SCC、自环、重复边和多分量确定性。

复杂度测试不得使用宽松耗时阈值。应结合受控比较调用、AST/源码结构证据和输出基数证明后处理边界。

## 8. 验收与停止条件

验收至少要求：11/11 目标测试、完整不少于 129/129、原 118/118、默认两次一致、`partial` 退出 0、`catalogs` 恰好 21 个 `OL001`、公开 API/CLI/标准库边界不变、正式非 README Markdown 仍为 0、缓存为 0、Git 范围和工作区卫生通过。

出现以下任一情况必须停止：需要新增 OL、改变表头或公开 API；需要第三方依赖或新模块；无法同时保留 R5 的围栏、HTML br 和 SCC 正向行为；记录身份无法在现有合同内限定且必须修改正式信息架构；原 118 项回归；需要外部写操作或修改全局决定。

## 9. 方案比较与最终选择

| 方案 | 优点 | 风险 | 裁决 |
|---|---|---|---|
| 在现有全局条件上继续补过滤 | 差异最小 | 状态键域继续耦合，组合漏报风险高 | 不采用 |
| 有界内部资格模型 | 在单模块内分离直接诊断和派生资格，能覆盖全部已知反例 | 需要重排内部状态登记流程 | 采用 |
| 拆分解析器与闭合引擎模块 | 长期结构清楚 | 扩大文件范围和回归面，违反本次最小返工原则 | 不采用 |

该设计不改变 Phase 1 信息架构或全局规格，只把 R5 已批准但未正确实现的记录级语义和 SCC 复杂度承诺落到可验证的内部边界。
