# P0-03 内容与治理模板设计说明

**状态：** 待书面审查
**设计日期：** 2026-07-11
**制定者：** 主会话
**前置基线：** P0-02 信息架构与 P0-02A AI 内容写作规则均已 `accepted`

## 1. 背景

P0-03 负责建立后续内容生产和项目治理使用的可复制模板。模板必须把 PRD、`AGENTS.md` 和 `AI-CODEX-RULES.md` 的要求转换为执行会话可以直接使用、主会话可以逐项验收的结构，同时避免把所有知识类型塞进同一套僵化目录。

原 Phase 0 实施计划列出九类模板，但此前已批准“可视化是一级内容”，候选规则整理也把 Draw.io 视觉细节迁移到 P0-03 图表模板或后续图形规范。用户于 2026-07-11 批准方案 A：P0-03 从九类扩展为十类，新增独立的 `图表模板.md`。

本设计通过后，新的 P0-03 实施计划将取代 `2026-07-10-Phase-0-治理与骨架-实现计划.md` 中 P0-03 的“九类模板”数量和文件清单；其余 Phase 0 范围、角色和提交原则保持不变。

## 2. 设计目标

1. 建立十类职责单一、可直接复制的模板；
2. 让知识章节根据 `type` 自适应，而非机械套用安装、实验、故障和配图模块；
3. 统一必填、可选、证据、状态和引用表达；
4. 把实验安全、云/GPU 成本、快照、回滚和清理固化为模板门禁；
5. 把故障恢复、永久修复和预防改进明确分开；
6. 让面试题与正文、实验、项目和故障证据建立来源关系；
7. 为 Mermaid 和 Draw.io 建立独立的图表生产与验收结构；
8. 让任务包、执行报告、验收报告和 ADR 保持执行、审查、验收职责分离；
9. 为 P0-04 自动校验器提供稳定、可机械识别的字段和占位符规则；
10. 不在模板中生产任何正式教学内容、实验结果或项目结论。

## 3. 非目标

- 不编写知识章节、实验、项目、故障案例、面试答案或正式图表；
- 不创建 Draw.io 源文件、Mermaid 正式图、代码、脚本或配置；
- 不建立 VitePress 站点；
- 不创建 P0-04 校验器；
- 不创建 P0-05 版本与来源基线；
- 不改变 PRD 的 21 个阶段、五个贯穿项目、Front Matter 字段、技术版本或状态集合；
- 模板不得硬编码 Rocky Linux、Kubernetes、Python 等动态产品版本；
- 不把模板扩展为第二份 PRD、AI 写作规则或企业编码规范。

## 5. 模板通用设计

### 5.1 直接复制原则

每个模板本身就是可复制骨架，不在外层再包一整段嵌套 Markdown 代码块。填写说明使用 HTML 注释，不影响复制后的渲染；必须删除的提示通过明确占位符识别。

每个模板开头必须说明：

- 模板用途和适用目录；
- 权威上游文件；
- 必填与可选模块；
- 复制后必须删除的说明；
- 与其他模板的关系；
- 禁止事项。

### 5.2 占位符协议

模板内只允许两类占位符：

```text
<必填：说明>
<可选：说明>
```

规则：

1. YAML 值中的占位符必须加引号，保证语法可解析；
2. 列表项可以使用带引号的占位符；
3. Markdown 标题或正文提示可以使用占位符；
4. 不使用 `TODO`、`TBD`、“以后补充”“这里填写”等模糊词；
5. P0-04 应允许上述两类标记出现在 `模板/`，但禁止残留在正式内容、实验、项目、故障、面试、图表说明和治理交付物中；
6. 可选模块不适用时整段删除，不能保留空标题；
7. 模板自身不得出现无法判定是必填还是可选的占位文本。

### 5.3 字段一致性

- 机器可读键保持英文，中文用于说明和值；
- 同一概念只使用一个名称，例如 `status`、验证证据、风险、未决问题；
- 日期使用 `YYYY-MM-DD`；
- 任务和治理状态只使用 PRD 第 15.2 节的固定集合；
- 内容章节状态只使用 `draft`、`reviewed`、`verified`、`deprecated`；
- 所有证据声明必须区分来源核验、静态检查、环境验证、综合实验、生产证据和未验证；
- 模板只引用已经存在的权威文件，不创建指向未来文件的失效 Markdown 链接。

## 6. 十类模板规格

### 6.1 知识章节模板

必须完整包含 PRD 第 5.2 节的 Front Matter 键：

```text
title
chapter
stage
type
difficulty
prerequisites
objectives
study_time
practice_level
keywords
versions
status
last_updated
```

模板必须标明九种允许的 `type` 和四种允许的内容 `status`，但不写死产品版本。

共享正文骨架为：

1. 本章目标与前置知识；
2. 技术定位；
3. 核心概念或关键规则；
4. 原理、流程或使用模型；
5. 实际场景或实践；
6. 验证与常见问题；
7. 学习检查；
8. 参考资料。

模板通过 HTML 注释给出 `concept/principle`、`service`、`platform`、`tool`、`language`、`cloud-service`、`hardware`、`methodology` 的扩展选择，不预先创建全部可选标题。背景、安装、配置、高可用、性能、安全、故障和面试均按类型与学习目标选择。

### 6.2 实验模板

必须包含：

1. 实验编号、等级、目标和完成标准；
2. 前置知识和关联正文；
3. 环境、拓扑、主机名、IP、版本与资源范围；
4. 账号、镜像、文件和变量来源；
5. 风险、影响范围、快照点和回滚方法；
6. 云资源或 GPU 的计费提示与预算条件；
7. 操作步骤、解释、验证命令和预期现象；
8. 常见错误或可控故障注入（适用时）；
9. 清理、释放、恢复与清理验证；
10. 实验总结、证据等级和扩展练习。

不得把实际未运行的命令结果写成已验证结果；示意输出与实际输出必须分开。

### 6.3 项目模板

必须包含：

- 项目编号、阶段、业务目标、学习者价值和非目标；
- 前置章节、实验和上游项目；
- 架构、组件、边界、数据流和版本入口；
- 里程碑、交付物、文件所有权和依赖；
- 环境、部署、配置、数据、权限和 Secret 策略；
- 发布、验证、回滚、备份和恢复；
- 日志、指标、链路、告警和故障演练；
- 安全、成本、容量、可用性和维护性取舍；
- 验收标准、证据、风险、未决问题和参考资料。

通用项目模板不得写入若依或 AI 项目的具体技术答案；五个项目的固定事实继续由 PRD 和未来项目任务包承载。

### 6.4 故障案例模板

必须按推理顺序包含：

1. 现象、影响范围和时间线；
2. 环境、版本、最近变更和已知约束；
3. 初步假设、优先级和安全边界；
4. 证据、命令、关键输出与排除过程；
5. 根因及根因证据；
6. 临时恢复；
7. 永久修复；
8. 验证、监控和回滚；
9. 预防措施与复盘改进；
10. 面试表达摘要和参考来源。

模板必须禁止随机命令堆叠、把重启当作根因修复、伪造现场和泄露敏感信息。

### 6.5 面试题模板

必须包含：

- 题目编号、类型、难度和对应岗位；
- 来源正文、实验、项目或故障案例；
- 前置知识和考察目标；
- 题目、回答结构和关键点；
- 原理追问、场景追问和设计追问；
- 常见错误、风险与权衡；
- 可以引用的真实实验或项目证据；
- 未做过生产操作时的诚实表达边界；
- 参考资料与复核状态。

模板不强制每题同时包含五种题型，也不允许脱离学习产物建立孤立背题库。

### 6.6 图表模板

模板必须覆盖：

- 图编号、标题、主要问题、读者结论和关联文档；
- 图类型以及 Mermaid、Draw.io 或表格的选择理由；
- 源文件、导出文件、正文引用路径和同名规则；
- 组件、分组、边界、流向、时序、状态和图例；
- Mermaid 节点 ID、主要节点数量和判断条件；
- Draw.io 布局方向、固定颜色语义、形状、线宽、字号和箭头一致性；
- 网络区域、信任边界、数据方向和风险标识（适用时）；
- 颜色非唯一编码、文字替代说明和正文读图说明；
- 来源、第三方商标/图标、许可证和原创归属；
- 版本范围、技术核验、导出验证和链接验证；
- 图源可编辑性、SVG 可渲染性和验收清单。

图表模板只规定一张图的生产合同，不生成空白 Draw.io 文件，也不强制每章固定图数。

### 6.7 任务包模板

必须落实 PRD 第 15.3 节十一项门禁，并在文件前部提供可直接复制的新会话启动指令。模板必须明确：

- 执行会话身份和固定工作目录；
- 必读文件和 AI 规则读取条件；
- 唯一目标、学习者价值、范围和非范围；
- 精确写入白名单和禁止事项；
- 前置、版本入口和官方资料要求；
- 交付物、验收标准、验证命令与预期；
- 提交拆分、执行报告、停止条件和完成回复；
- 不得自行 push、创建 PR、发布或修改 GitHub 设置；
- 执行会话只能建议状态，不得自验收。

### 6.8 执行报告模板

必须包含：

- 任务、执行角色、日期、建议状态；
- 文件清单和范围确认；
- 交付提交哈希，报告提交自身哈希的记录规则；
- 验收标准到证据的逐项映射；
- 实际命令、退出码和关键输出；
- 来源、版本、验证环境和证据等级；
- 异常、权限请求、风险和未决问题；
- 远程操作确认；
- 不得宣布 `accepted`。

### 6.9 验收报告模板

必须包含：

- 审查对象、提交链、验收角色和日期；
- 规格范围、文件边界、技术证据、教学质量、来源版本、链接格式和集成影响；
- 主会话独立复验，不复制执行报告结论；
- 独立审查意见及主会话裁决；
- `accepted`、`rework` 或 `blocked` 的明确结论；
- 返工差距、允许范围和复验条件；
- 风险、未决问题、后续任务门禁和远程授权边界。

### 6.10 决策记录模板

采用 ADR 结构，必须包含：

- 编号、标题、状态、日期、制定者和影响范围；
- 背景、约束和决策问题；
- 至少两个真实可行选项及权衡；
- 决定与理由；
- 正面、负面和中性后果；
- 实施与迁移影响；
- 复审触发条件；
- 被替代、废弃和关联决策；
- 参考证据。

## 7. 模板 README

`模板/README.md` 是模板入口，必须说明：

- 十类模板索引及各自所有权；
- 复制与删除 HTML 注释的方法；
- `<必填：说明>`、`<可选：说明>` 占位符协议；
- 正式内容不得残留占位符或空标题；
- AI 规则、PRD、模板和任务包的职责层级；
- 哪些模板使用 Front Matter，哪些使用 Markdown 字段；
- 版本、来源、证据和敏感信息边界；
- 模板变更的复验要求。

## 8. 执行和审查流程

### 8.1 P0-03 执行

独立执行会话一次性创建模板 README 和十类模板，不编写正式内容。模板提交与执行报告必须是两个独立提交：

```text
docs: add project content templates
docs: add P0-03 execution report
```

### 8.2 P0-03B 独立审查

模板是跨模块公共基线，影响所有后续内容任务，属于高影响治理成果。P0-03 执行提交后，必须由未参与模板创建的独立会话进行 P0-03B 审查。审查会话只创建独立审查报告，不修改模板。

独立审查至少覆盖：

- 十类模板的职责和字段完整性；
- 知识模板是否自适应；
- 实验、图表、故障和治理模板的安全与证据门禁；
- 占位符是否可机械识别；
- 是否存在第二份 PRD、重复规则或硬编码版本；
- 模板之间的字段一致性和链接；
- 执行报告与 Git 事实是否一致。

### 8.3 主会话验收

只有 P0-03 执行和 P0-03B 独立审查均无阻塞、无必须修改，主会话才能将模板基线标记为 `accepted`。P0-04 在此之前保持 `planned`。

## 9. 自动验证设计

P0-03 任务包必须提供不落盘的 PowerShell 检查，至少验证：

1. 模板目录文件总数为 11；
2. 除 README 外模板数为 10；
3. 文件集合与白名单完全相等，缺失和越界均为 0；
4. 知识章节模板包含 13 个 Front Matter 键、九种 `type` 和四种内容 `status`；
5. 知识模板包含八项共享骨架，并明确可选模块按类型选择；
6. 实验模板命中风险、成本、快照、回滚、验证、清理和资源释放；
7. 故障模板命中临时恢复、永久修复、根因证据、回滚和预防；
8. 面试模板命中来源、追问、项目证据和诚实表达；
9. 图表模板命中 Mermaid、Draw.io、图编号、源文件、SVG、可访问说明、许可证和验证；
10. 任务包模板命中 PRD 15.3 的十一项门禁和完整启动指令；
11. 执行、验收和 ADR 模板字段完整；
12. 十类模板都包含用途、必填/可选说明和禁止事项；
13. 占位符只使用两种批准格式；
14. `TODO`、`TBD`、模糊占位词和敏感信息命中为 0；
15. 不包含固定产品补丁版本、正式教学答案或伪造验证结果；
16. 所有 Markdown 相对链接有效；
17. `git diff --check` 通过；
18. 模板提交和执行报告提交范围准确，最终工作区干净。

P0-04 将在模板基线接受后把其中适合机械化的规则转成 Python 标准库校验器；P0-03 不创建临时或正式校验脚本。

## 10. 人工审查设计

以下内容不能只靠关键词搜索：

- 模板是否真正可直接复制；
- 知识模板是否会诱导执行会话机械填充空模块；
- 占位符和 HTML 注释是否清楚且不互相冲突；
- 实验模板是否能让零基础学习者安全恢复环境；
- 图表模板是否降低认知负担，而不是装饰性规范；
- 故障模板是否体现证据驱动排查；
- 面试模板是否阻止虚构生产经验；
- 治理模板是否保持执行、审查和最终验收分离；
- 各模板是否只承担一个职责；
- 模板是否重复或改写 PRD 和 AI 规则。

## 11. 验收标准

1. 十类模板和 README 全部存在，路径精确；
2. 图表模板作为第十类独立模板存在；
3. 模板通用占位符、字段和证据规则一致；
4. 知识章节模板完整、自适应且没有固定安装模块；
5. 实验、项目、故障、面试和图表模板符合本设计；
6. 四类治理模板符合 PRD 和 AGENTS；
7. 模板没有正式内容、动态版本、敏感信息或伪造证据；
8. 自动检查全部通过；
9. 执行报告与提交事实一致；
10. 独立审查无阻塞、无必须修改；
11. 主会话完成独立复验和最终验收；
12. 全过程未执行未经授权的远程写操作。
