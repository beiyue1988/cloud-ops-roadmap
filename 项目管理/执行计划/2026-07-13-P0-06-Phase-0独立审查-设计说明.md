# P0-06 Phase 0 独立审查设计说明

**批准方案：** 阶段门综合审查（方案 A）

**前置状态：** P0-01 至 P0-04 及其已验收子任务均为 `accepted`；P0-05 为 `submitted`

## 1. 背景

Phase 0 已建立项目治理、信息架构、AI 内容规则、内容与治理模板、仓库校验器，以及版本与来源基线。各高风险任务已经经过专项执行、返工、独立审查和主会话验收，但 Phase 0 仍需要一次跨交付物的独立阶段审查，证明这些成果在同一仓库状态下能够共同工作，并且具备进入 Phase 1“导航与完整大纲”的条件。

P0-06 不是把所有历史专项审查重新执行一遍，也不是只检查当前文件是否存在。它审查固定 Phase 0 交付快照、关键提交链和当前集成状态：复用已有专项审查结论，同时重新运行能够证明“当前没有漂移”的自动验证和跨文件人工检查。

本设计取代 `2026-07-10-Phase-0-治理与骨架-实现计划.md` 中 P0-06 的简略规格。原任务目标、单报告白名单、只读审查、三种建议结论和主会话最终验收责任保持不变；新增固定审查对象、证据分层、八个审查域、动态版本复核、历史链抽查、问题分级和 Phase 1 解锁门。

## 2. 目标

1. 独立确认 Phase 0 固定交付快照的提交范围、来源和历史链可追溯；
2. 证明治理文件、信息架构、规则、模板、校验器、版本基线和项目记忆之间没有阻塞性冲突；
3. 重新运行当前完整自动验证，证明已经验收的工具和 Markdown 集成没有漂移；
4. 重新核验高变动版本事实、远程连通性、许可证与安全边界；
5. 识别必须返工、非阻塞建议和主会话最终验收动作；
6. 给出 `recommend-accept`、`recommend-rework` 或 `blocked`，但不自行接受 Phase 0；
7. 为主会话形成最终 Phase 0 验收报告和启动 Phase 1 设计提供独立证据。

## 3. 非目标

- 不修改、修复或重写任何被审文件；
- 不创建 P0-01 至 P0-05 的新执行报告或替代历史报告；
- 不重复 P0-04 已验收的完整历史红灯重建和所有异常注入探针；
- 不安装 Rocky Linux、Kubernetes、containerd、Calico、VitePress 或 RuoYi-Vue；
- 不生成学习路线、技术索引、章节树、知识正文、实验、项目、图表或脚本；
- 不更新项目记忆、任务看板、ADR 状态、版本基线或许可证；
- 不创建返工任务包、Phase 1 计划或最终验收报告；
- 不执行 push、Pull Request、网站发布、制品发布或 GitHub 设置修改；
- 不把独立审查报告的建议结论写成项目最终 `accepted`。

## 8. 人工审查矩阵

审查报告必须完成 24 项人工检查，每个审查域三项：

| 域 | 人工检查 |
|---|---|
| Git 与治理 | 角色分离真实；状态与报告链一致；返工历史有关闭链 |
| 信息架构 | 双导航单一源成立；目录职责无冲突；无提前正文且 Phase 1 可增量进入 |
| AI 规则 | 权威层级无冲突；规则覆盖研究/写作/安全/证据；停止升级机制可执行 |
| 模板 | 十类模板职责清楚；自适应/必填边界合理；高风险字段与报告责任分离 |
| 校验器 | 当前对象无漂移；自动规则没有越权代替人工审查；README 与行为一致 |
| 版本与来源 | 锁定/阶段门/证据语义准确；动态事实新鲜；ADR 取舍和负面后果真实 |
| 许可与安全 | 双许可证边界一致；第三方与敏感信息处理合规；远程/云/GPU 授权未扩大 |

每项结果只能为：`通过`、`需说明`、`不通过`。`需说明` 必须给出影响、责任人和最晚门禁；不能用它隐藏必须修改。

## 9. 问题分级与建议结论

### 9.1 问题分级

| 分级 | 定义 | 处理 |
|---|---|---|
| `blocking` | 无法建立固定审查对象、发现真实敏感信息、关键证据损坏或安全边界失效 | 仓库缺陷时 `recommend-rework`；外部条件导致无法审查时 `blocked` |
| `must-fix` | Phase 0 完成条件、上层规格或当前版本事实不满足，但可由明确返工修复 | `recommend-rework`，给出精确文件、期望修复和复验命令 |
| `suggestion` | 不影响 Phase 0 接受，但能提升可维护性或后续执行质量 | 可以 `recommend-accept`，必须指定后续责任和门禁 |
| `passed` | 有独立证据支持的通过项 | 记录证据，不产生修改要求 |

主会话必然需要完成的验收动作——创建最终验收报告、更新任务看板、补充项目记忆中的 ADR-0003 至 ADR-0005 摘要——不算 P0-06 交付缺陷；审查报告必须将它们列为 `recommend-accept` 后的接受前动作。若现有项目记忆与 ADR 存在相反结论，则属于 `must-fix`。

### 9.2 建议结论

报告结论只允许：

- `recommend-accept`：无 `blocking`、无 `must-fix`，自动门禁通过，24 项人工检查没有不通过；可以带非阻塞建议和主会话收尾动作；
- `recommend-rework`：存在仓库内可修复的 `blocking` 或任一 `must-fix`；必须给出差距、文件边界、修复期望和复验方法；
- `blocked`：外部权限、网络、来源不可用或审查对象无法确定，导致关键证据无法取得；必须给出解除条件。

审查会话不得使用 `accepted`、`rework` 或 `passed` 代替上述阶段建议结论，也不得修改任务看板。

4. 14 组自动验证的命令、退出码、关键输出和统计；
5. 八个审查域、24 项人工检查和直接证据；
6. 动态来源、核验日期、支持范围和限制；
7. `blocking`、`must-fix`、`suggestion`、`passed` 数量与明细；
8. 主会话最终验收前必须完成的收尾动作；
9. 风险、未决问题、外部阻塞和解除条件；
10. 远程只读操作、未授权写操作和临时文件清理确认；
11. 建议结论，不宣布 Phase 0 或任何任务 `accepted`。

报告必须区分：现有交付事实、审查会话新鲜复验、对历史报告的引用、推论和未验证边界。不能复制执行报告结论代替独立判断。

## 12. Phase 0 最终接受与 Phase 1 解锁

P0-06 给出 `recommend-accept` 后，主会话仍需独立完成：

1. 核对 P0-06 报告提交范围、证据和独立性；
2. 复验关键自动门禁和所有审查意见；
3. 裁决 `suggestion` 与主会话收尾动作；
4. 创建 Phase 0 最终验收报告；
5. 将 P0-05、P0-06 和 Phase 0 相关状态更新为正式接受；
6. 将 ADR-0003 至 ADR-0005 加入项目记忆索引和活跃决策摘要；
7. 明确保留 GPU/模型组合作为 AI Infra 阶段门；
8. 只解锁 Phase 1 的设计与任务包制定，不直接生成批量正文；
9. 继续禁止未授权的 push、Pull Request、发布和 GitHub 设置修改。

若 P0-06 建议返工，主会话按问题归属创建精确返工任务；不得让审查会话直接修复。返工和复审通过前，Phase 1 保持不可启动。

## 13. 停止条件

- 审查会话不独立；
- 工作区不干净、分支或远程不符；
- 固定交付截止不是当前 HEAD 祖先；
- 固定截止后的差异超出 P0-06 治理文件和任务看板；
- 需要修改报告之外的文件才能继续；
- 发现真实敏感信息或无法保证临时数据安全清理；
- 自动测试、校验或版本来源失败，且无法在只读范围确定是交付缺陷还是审查工具缺陷；
- 远程连通或关键官方来源在三次合规尝试后仍无法核验；
- 任务包、设计、计划、PRD、AI 规则或固定对象存在无法解释的冲突；
- 无法形成单一报告提交或需要改写被引用历史。

停止不等于审查失败。仓库内可定位缺陷使用 `recommend-rework`；外部条件使关键证据无法取得时使用 `blocked`，并写明恢复条件。
