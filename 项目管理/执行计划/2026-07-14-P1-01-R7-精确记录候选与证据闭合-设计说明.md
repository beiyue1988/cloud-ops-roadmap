## 3. R6 退回问题

### 3.1 跨所有者键污染

`_ProjectRecordExclusion` 保存 `owner`，但匹配只使用来源侧和原始/预期键。项目 01 总视图中的错误 `PRJ-02-M01` 会按全局字符串屏蔽项目 02 合法定义缺失于总视图的真实 `OL011`。

### 3.2 无稳定身份仍推导 D=V

总视图里程碑无法解析出稳定后缀时，原始键和预期键都为空，但整张视图仍保持可比较。错误行因此被放大为项目侧“未出现在贯穿项目视图”的派生 `OL011`。

### 3.3 缺失分隔行吞掉首行诊断

解析器无条件把表头下一行当作分隔行。分隔行实际缺失时，首个列数正确的数据行被跳过，无法执行字段格式、非空或本行键域诊断。

### 3.4 SCC 结构证明不足

当前 SCC 实现已采用一次归一化和线性装桶，但现有测试只检查 `reverse` 赋值后的直接 `sorted(...)` 调用。它不能阻止 `.sort()`、辅助排序、堆或重复最小值扫描，也没有受控比较次数证据。

## 4. 方案比较与决定

### 4.1 方案 A：局部增加 owner 等值判断

优点是差异最小。缺点是章节引用只携带锚点键，无法同时表达“错误项目记录自身的原始键引用应隔离”和“另一项目对同名键的真实缺口应保留”。单纯 owner 等值会修复一种漏诊并重新引入另一种级联，因此不采用。

### 4.2 方案 B：精确记录候选模型

行解析后形成带来源侧、记录所有者、原始键、预期键、章节关系和身份稳定性的内部候选。合格候选登记状态；非法候选只参与有上下文的自身级联隔离。该方案能同时覆盖四项退回问题，且不改变外部接口，采用。

### 4.3 方案 C：任一坏行关闭整表闭合

实现简单，但会重现 R5 的全表失资格问题，吞掉无关项目的真实 `OL011`，不采用。

## 5. 内部模型

实现可以按现有命名做最小调整，但必须具备等价于以下结构的语义：

```text
ProjectRecordCandidate
  side: view | project
  owner: 01..05 | unknown
  raw_key: PRJ-NN-MCC | unknown
  expected_key: PRJ-owner-MCC | unknown
  chapters: parsed chapter set | unavailable
  path: repository-relative path
  line: physical line
  closure_fields_valid: bool
  identity_stable: bool
```

闭合查询还必须携带等价于以下内部上下文，名称可按现有代码最小调整：

```text
AnchorReference
  anchor: CP | LAB | PRJ key
  path: repository-relative path
  line: physical line
  source: chapter | view | other
  owner: project owner | unknown
  related_chapters: parsed chapter set | unavailable
```

`closure_fields_valid` 只表示所有者、里程碑键和章节集合等闭合相关字段合格；能力结果、证据类型等非闭合文本继续产生直接诊断，但不得无关扩大闭合失资格。结构合格项目表中的每条记录都形成候选：合格候选登记有效状态，非法候选只作为自身级联证据；两个隔离谓词必须防御性拒绝 `closure_fields_valid=True` 的候选。`AnchorReference.source` 冻结为 `chapter`、`view` 或 `other`，只增加内部查询上下文，不改变公开 API 或诊断格式。

候选承担三项相互分离的责任：

1. 保存直接诊断所需的原始行上下文；
2. 决定该行能否登记为有效锚点、项目映射、视图记录或位置；
3. 为闭合阶段提供有作用域的自身级联隔离证据。

非法候选不得进入 `anchor_definitions`、`project_mappings`、`project_locations` 或有效 `project_view_records`。不得继续维护只按全局字符串匹配的排除集合。

## 6. 数据流与资格规则

### 6.1 两阶段受控表处理

所有受控表先完成结构识别，再对保留下来的行候选执行直接验证：

1. 识别表头、分隔行、数据区边界和数据物理位置；
2. 对每个结构可解析的数据候选执行字段格式、非空、本行键域和引用语法验证；
3. 只有相应表级资格合格时，才执行连续性、顺序、重复、覆盖、状态登记和集合闭合。

表结构失败仍禁止表级推导，但不能阻止结构可解析行产生直接诊断。

### 6.2 分隔行分类

- 合法分隔行：数据从下一行开始，首行位置为 1。
- 非法但仍呈分隔符形态的行：报告分隔行 `OL002`，不把该行当成数据；下一结构可解析行位置为 1。
- 列数正确且明显不是分隔符形态的行：视为“分隔行缺失后的首个数据候选”；同时报告分隔行 `OL002`，保留该行并赋予位置 1。
- 列数或边界不可解析的行：只报告结构错误，不伪造数据候选。

“分隔符形态”必须由仅含现有分隔语法字符的单元格机械判定，不能依赖具体表内容或自然语言猜测。

### 6.3 项目 README 侧

合格记录继续登记锚点定义、项目映射和位置。非法记录的隔离只能覆盖以下关系：

- 对预期键的查询，所有者必须与候选所有者一致；
- 对错误原始键的引用，只有来源明确为 `chapter`，且引用章节属于该非法候选已经成功解析的章节集合时，才视为错误行自身级联；
- 章节集合不可用但预期键稳定时，可以隔离同一所有者、同一预期键的闭合，不得隔离错误原始键内嵌项目号所属项目的其他记录。

因此，项目 01 错用 `PRJ-02-M02` 且列出章节 `08.01` 时，可以隔离 `08.01` 章节引用对该错误行的自身级联；项目 02 的合法 `PRJ-02-M02` 定义、总视图记录或其他章节关系仍正常参与闭合。即使合法总视图记录也列出 `08.01`，其 `source=view` 引用仍不得被项目 01 的错误原始键解释。

### 6.4 项目总视图侧

非法总视图记录只允许用“来源侧 + 记录所有者 + 预期键”隔离该所有者的 D=V 缺口。错误原始键不能屏蔽其内嵌项目号所属项目的合法映射。

当项目列有效且里程碑保留 `MCC` 后缀时，预期键为 `PRJ-owner-MCC`。当无法得到稳定后缀或记录所有者时，候选标记为身份不稳定，停止该总视图的 D=V 集合比较。停止 D=V 不影响：

- 当前行直接诊断；
- 项目里程碑到章节的反向引用检查；
- 章节到项目里程碑的列回检查；
- 检查点、实验和其他表的闭合。

### 6.5 合法状态优先

闭合查询先使用合法定义和合法映射。只有在合法对应项确实不存在时，才检查非法候选是否精确解释该缺口。非法候选不能覆盖、替代或使合法候选失资格。

## 7. SCC 证据设计

保留当前非递归 SCC 实现和一次归一化 + 线性装桶架构。R7 冻结 `_catalog_cycles` 相对 R6 的完整 AST，不修改该函数；其 `ast.dump(..., include_attributes=False)` 的 UTF-8 SHA-256 固定为 `705deb57eb4a09333016b64c8d451d2df363fb74daa073fdafc000033d98d604`。

现有 `test_catalog_cycles_has_no_post_normalization_comparison_sort` 必须强化，但不新增测试方法：

1. 使用至少 256 个反序插入、可统计比较次数的 ID 构造稀疏单环；
2. 在固定 Python 3.11.15 下，每次运行的受控比较计数必须恰好为 255，证明 256 个反序 ID 只在规范节点顺序建立阶段完成一次比较排序；
3. 断言完整函数 AST 摘要等于上述 R6 固定值，同时保留归一化边界后的直接禁止调用检查；任何新增内部辅助调用、`sorted(...)`、`.sort()`、堆式排序或重复 `min/max` 扫描都会改变摘要并失败；
4. 断言返回一个分量、成员数量与唯一性正确、最小 ID 定位正确；
5. 重置计数后重复运行，结果和比较边界一致；
6. 禁止使用耗时阈值。
