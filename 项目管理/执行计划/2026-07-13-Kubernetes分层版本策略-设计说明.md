# Kubernetes 分层版本教学策略设计说明

**状态：** 方案已批准，书面规格待用户确认

**设计日期：** 2026-07-13

**制定者：** 主会话

**批准方案：** 分层版本策略（方案 1）

**关联决策：** [ADR-0004：采用 Kubernetes 1.36 教学基线](../决策记录/ADR-0004-Kubernetes-1.36教学基线.md)

**关联审查：** [Phase 0 独立审查报告](../验收记录/Phase-0-独立审查报告.md)

## 1. 背景

项目此前通过 ADR-0004 将 Kubernetes 1.36 minor、kubeadm、containerd 2.3 LTS、CRI v1、cgroup v2/`systemd` driver 和 Calico 3.32 设为主教学实验组合，同时将 Kubernetes 1.31 降为 `reference-only`。

P0-06 独立审查完成后，项目进一步澄清了“主教学基线”的含义：主实验环境需要唯一、受支持并可复验，但知识讲解不能假设所有企业都运行同一 minor，也不能把“使用当前受支持版本”误写成“只有最新版本值得学习”。企业存量环境、教材生态和升级路径具有教学价值，但不应因此让零基础学习者维护多套完整集群，或把 EOL 版本重新提升为默认环境。

本设计将 Kubernetes 版本拆成四个教学角色，保留 ADR-0004 的主基线，不修改已经审查的版本表，也不为 Phase 0 新增第二套实验环境。

## 3. 目标

1. 为零基础学习者保留唯一、清晰和可复验的主实验环境；
2. 将 Kubernetes 核心概念与具体 minor 解耦，减少不必要的版本绑定；
3. 让学习者理解企业环境存在版本滞后，而不复制第二套完整课程；
4. 解释 EOL、版本偏差、API 弃用和升级风险，不把旧版本包装成新部署建议；
5. 保持 patch 值集中维护，正文不分散复制动态版本；
6. 在阶段 14 实验进入 `ready` 前重新核验当时的支持期、组件交集和环境可复现性。

## 4. 非目标

- 不把 Kubernetes 1.35 或 1.31 新增为第二套完整实验环境；
- 不把 Kubernetes 1.34 设为主环境或兼容基线；
- 不把 1.31.6 设为项目级默认、推荐安装版本或通用升级起点；
- 不承诺覆盖企业正在使用的全部 Kubernetes minor；
- 不为匹配旧教程而保留已失效的命令、仓库、API 或安全做法；
- 不在 Phase 0 安装 Kubernetes、containerd 或 Calico；
- 不改变 CKA/CKAD 等认证的版本选择规则，认证任务仍以考试当期官方环境为准；
- 不修改 P0-06 固定审查对象或重写其历史报告。

## 5. 四层版本角色

### 5.1 概念教学层：版本中立

适用于：架构、控制循环、声明式 API、Pod、Deployment、Service、ConfigMap、Secret、调度、存储、网络、RBAC、可观测性和故障排查方法。

规则：

- 默认解释稳定概念，不在标题和每段文字重复写 `1.36`；
- 某能力、字段、默认行为或 API 在不同 minor 中有差异时，必须标注引入、弃用、移除或适用版本；
- 只使用主实验环境实际验证过的命令和清单作为可复制操作；
- 二手教程可用于发现学习难点，不能成为动态命令、兼容性或安全结论的唯一证据；
- 不因兼容旧教程而教授已经失效的 API。

### 5.2 主实验层：Kubernetes 1.36.x

定位：唯一默认安装、命令执行、项目部署、故障注入和环境验证基线。

规则：

- 锁定 minor `1.36` 和 kubeadm；
- patch 不写死在章节中，从 `版本记录/软件版本基线.md` 读取；
- 阶段 14 首个集群实验进入 `ready` 前核验当时最新安全 patch；
- 配套组合继续使用 containerd 2.3 LTS、CRI v1、cgroup v2/`systemd` driver 和 Calico 3.32；
- 官方来源兼容只标记 `source-verified`，实际集群通过后才可标记 `environment-verified`；
- 若 1.36 已进入维护模式、EOL、组件交集失效或环境无法复现，必须停止签发实验任务并由替代 ADR 选择新的受支持 minor。

“主实验层”只表示仓库默认验证对象，不表示企业只能使用 1.36，也不表示所有概念都是 1.36 独有。

### 5.3 企业兼容层：Kubernetes 1.35.x

定位：用于说明企业可能落后一个 minor、托管平台升级节奏、版本偏差、组件矩阵和迁移判断。

规则：

- 作为比较与兼容参考，不创建第二套从安装到项目的完整实验链；
- 只有具体知识点存在可观察差异时才加入 1.35 对照；
- 具体 patch 在使用该案例的任务进入 `ready` 前核验，不在本设计中锁定；
- 可以使用官方文档、Release Notes、版本偏差策略和组件矩阵说明差异；
- 若需要实际运行 1.35，必须由独立实验任务说明学习价值、资源、快照、清理、版本组合和验收证据；
- 1.35 EOL 后自动降为历史参考，不继续称为企业兼容受支持线。

企业采用率、博客数量或视频数量如果没有可复验数据，只能作为选题线索，不能单独决定主环境。

### 5.4 历史升级层：Kubernetes 1.31.14

定位：解释 EOL、旧集群维护边界、逐 minor 升级、API 弃用、CNI/CSI 兼容和技术债务。

规则：

- 使用 1.31 系列最终 patch `1.31.14` 作为分支历史代表，不使用中间 patch `1.31.6` 作为通用默认；
- 保持 `reference-only`，明确不再接收安全更新与缺陷修复；
- 不允许面向公网、不允许写成生产新部署方案、不允许省略风险提示；
- 默认以解释、清单差异或隔离演示为主，不要求每位零基础学习者安装；
- 真正升级实验必须遵守 Kubernetes 逐 minor 升级约束，不能把 1.31 直接升级到 1.36；
- 如果某个真实企业案例明确固定在 1.31.6，可以在案例内记录该事实，但必须说明它只是案例起点，并将升级前的同 minor 修补评估纳入方案。

## 6. 学习路径中的使用方式

学习者看到的顺序为：

1. 先学习版本中立的核心概念；
2. 在唯一 1.36.x 主环境中完成可复制实验；
3. 在相关知识点旁理解 1.35 的企业兼容差异；
4. 在升级与技术债务专题中理解 1.31 的 EOL 和迁移风险。

不得先向零基础学习者展示三套安装流程再解释 Kubernetes 核心概念。版本比较服务于工程判断，而不是制造额外记忆负担。

## 7. 内容与实验标注规则

### 正文

- 通用概念不添加版本标签；
- 存在差异时使用“适用版本”“在 1.35/1.36 中的差异”“自某版本起”等明确表述；
- “当前”“最新”“支持”必须带核验日期并链接版本入口；
- 不在正文复制当前 patch 表。

### 实验

- Front Matter 的 `versions` 记录实际验证组合；
- 主实验只允许使用版本入口的当前 1.36 patch；
- 兼容/历史实验在标题、风险和清理部分显式标注用途；
- 模拟输出与实际验证输出分开；
- EOL 环境必须限制网络暴露、保存快照并在实验后销毁。

### 面试与企业案例

- 区分“官方支持状态”“企业实际存量”“本人实际操作证据”；
- 不把在 1.36 做过的实验表述为维护过 1.31 生产集群；
- 回答版本问题时先说明选择维度，再说明本项目主实验与历史参考的边界。

## 8. 治理影响

### 8.1 保持不变

- ADR-0004 继续有效，不被本设计替代；
- Kubernetes 1.36 继续作为主实验 minor；
- Kubernetes 1.31 继续为 `reference-only`；
- P0-05 版本基线的 `locked/phase-gated/reference-only` 统计不在本设计阶段修改；
- P0-06 对固定截止的审查结论保持有效。

### 8.2 后续落实

- Phase 0 最终验收记录本策略是 P0-06 固定截止后的用户批准解释，不冒充 P0-06 已审内容；
- 项目记忆在收尾时摘要 ADR-0004，并注明“主实验不等于唯一学习版本”；
- Phase 1 的章节树和学习路线为企业版本判断、EOL 与升级路径预留位置；
- 阶段 14 的实施计划按当时官方事实核验 1.36 patch 和配套组合；
- 若后续需要把 1.35 变成实际验证环境或改变主 minor，必须创建独立任务和替代 ADR。

### 8.3 不需要的治理动作

本策略没有改变主 minor、运行时、CNI、实验平台或许可证，因此不要求：

- 返工 P0-05 版本基线；
- 重做 P0-06 全量独立审查；
- 新增第二套版本矩阵或多版本 CI；
- 立即创建新的 Kubernetes 实验。

主会话在 Phase 0 最终验收时仍需复验本设计没有修改 P0-06 固定对象，并明确其适用边界。

## 9. 失败与停止条件

后续任务遇到以下情况必须停止并交由主会话裁决：

- 把 1.35 或 1.31 当作默认完整实验环境，但没有独立任务授权；
- 把 1.31.6 写成项目推荐安装版本；
- 主实验任务进入 `ready` 时 1.36 已不满足支持期或组件交集；
- 版本差异结论只有二手教程支持，无法取得官方直接证据；
- EOL 实验缺少隔离、快照、回滚、资源清理或风险提示；
- 需要改变 ADR-0004 的主 minor 或配套组件；
- 认证考试版本与项目主实验版本不同，却试图修改全局基线来追随一次考试。

## 10. 验收标准

1. 版本中立概念、1.36 主实验、1.35 企业兼容和 1.31 历史升级四个角色互不混淆；
2. 唯一默认安装环境仍为 Kubernetes 1.36.x；
3. 1.35 不形成第二套完整课程，具体 patch 在使用前核验；
4. 1.31 使用最终 patch 1.31.14 代表分支历史，1.31.6 只允许出现在明确案例中；
5. EOL、支持期、来源核验和环境验证术语准确；
6. 主实验不被描述为唯一企业版本；
7. 概念正文不机械绑定版本，真实差异必须标注；
8. 动态 patch 继续集中维护，不分散写入章节；
9. 本设计不修改 ADR-0004、版本基线或 P0-06 固定审查对象；
10. Phase 1 和阶段 14 的后续门禁、责任和停止条件明确；
11. 不新增多版本 CI、第二套完整实验或认证专用基线；
12. 任何未来主 minor 变化都必须通过替代 ADR 和独立验证。

```text
主实验环境：Kubernetes 1.36.x

历史升级参考：Kubernetes 1.31.14
```
