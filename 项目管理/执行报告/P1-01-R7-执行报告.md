## 4. 实现与合同闭合

### 4.1 分隔行分类

- 合法分隔行：数据从下一行位置 1 开始。
- 全部单元格只含可选冒号和至少一个连字符的非法分隔符形态：只报分隔 `OL002`，不伪装成数据。
- 列数正确且非分隔符形态的表头下一行：保留为位置 1 数据候选，继续执行直接字段诊断。
- 任何结构失败仍使整表退出连续性、覆盖、登记和闭合；后续行不按压缩下标重排。

### 4.2 候选、引用上下文与合法状态优先

- `_ProjectRecordCandidate` 保存 `side`、`owner`、`raw_key`、`expected_key`、`chapters`、`path`、`line`、`closure_fields_valid` 和 `identity_stable`。
- `_AnchorReference` 保存 `anchor`、`path`、`line`、`source`、`owner` 和 `related_chapters`；来源固定使用 `chapter`、`view`、`other`。
- 结构合格表中的每条项目记录形成候选；只有闭合字段合格的候选登记有效定义、映射、位置和视图记录。
- `closure_fields_valid` 只覆盖 owner、里程碑键和章节集合；能力结果、证据类型等文本错误仍产生直接诊断，但不无关扩大闭合失资格。
- 两个隔离谓词均防御性拒绝 `closure_fields_valid=True`；闭合查询始终优先合法定义和合法映射。

### 4.3 owner、章节关系和不稳定身份

- 项目 README 侧的预期键隔离必须匹配候选 owner；错误 raw key 只在 `source=chapter` 且引用章节与候选已解析章节集合相交时解释自身级联。
- 因此项目 01 的 `08.01` 自身关系不再吞项目 02 的 `13.01` 缺定义/未列回，也不吞同为 `08.01` 的 `source=view` 合法引用。
- 总视图 D=V 缺项只由 view 候选的 owner + expected key 解释；错误 raw key 不再屏蔽其内嵌项目号所属项目。
- owner 或 `MCC` 后缀不稳定时，只停止总视图 D=V；当前行直接诊断及项目—章节双向闭合继续。

### 4.4 SCC 证据
