## 6. 根因与修复

### 6.1 直接诊断与表资格

- 根因：`structure_valid` 同时控制逐行直接验证、表级顺序/集合检查和状态登记；表中任一坏行使所有可解析行被跳过。
- 修复：`_TableRow` 保存数据区物理位置；受控表在表头、分隔或数据行结构失败时仍保留列数可解析行。所有受控表先执行字段格式、非空和本行键域直接验证；只有表结构合格时才运行连续性、覆盖、重复、顺序、集合、计数和状态登记。
- 证据：三个结构目标方法分别证明后续 `00.02`、合法 M02 和后续 CP 行不会按压缩下标产生伪诊断。

### 6.2 项目记录级资格

- 根因：`project_view_milestones_comparable`、`invalid_project_numbers` 和 `invalid_project_milestones` 是全局或整项目开关，一条坏记录会关闭无关记录的闭合。
- 修复：项目总视图和项目 README 逐记录判定结构、键与章节集合资格；其他合格记录继续参与 D=V、逐里程碑集合及双向映射。项目 README 整表结构失败只进入对应项目侧表域；项目总视图整表结构失败只停止该视图 D=V。
- 证据：PRJ-01 三类非法记录与独立 PRJ-05 缺项组合均同时保留直接错误和 PRJ-05-M01 `OL011`；非法兄弟记录不再关闭合法 M01。

### 6.3 精确键域

- 根因：项目行在验证归属和连续性前就调用 `_record_anchor_definition`，并用错误锚点内嵌项目号或全局字符串集合决定资格。
- 修复：失资格记录保存来源侧、记录所有者、原始错误键和可确定的预期键；只有表结构、项目归属、物理位置连续性和章节集合都合格的记录才登记 `anchor_definitions`、`project_mappings` 与 `project_locations`。隔离匹配仅在缺少有效同名定义/映射时抑制本错误记录的级联；另一项目的合法同名记录仍正常闭合。
- 证据：项目 01 错用 `PRJ-02-M02` 只保留本行 `OL010`；项目 02 合法定义同名键时无重复定义，并保留其独立真实 `OL011`。

### 6.4 检查点定义侧隔离

- 根因：检查点坏表未登记任何 CP，闭合阶段仍把章节 CP 引用作为普通缺失定义处理。
- 修复：检查点表单独记录定义侧比较资格；坏表继续报告结构和可解析行直接错误，但跳过缺失定义派生。合法表继续登记定义并检查 CP-00–CP-20、顺序、格式和缺项。
- 证据：目标方法无“CP-00 不存在”派生错误；原合法缺 CP、错序和错格式测试继续通过。

### 6.5 SCC 线性后处理

- 根因：Kosaraju 两遍遍历后又比较排序分量成员和分量列表。
- 修复：邻接归一化阶段一次建立 `node_order` 与 `node -> rank`；第二遍记录 `node -> component`、规模和自环状态；随后按 `node_order` 一次扫描装桶并按首次遇到的循环分量输出。
- 证据：归一化后 AST `sorted` 调用 0；递归、自调用、`setrecursionlimit` 和 `RecursionError` 兜底均为 0；每个循环分量仍只返回一条按最小 ID 定位的 `OL007`。
