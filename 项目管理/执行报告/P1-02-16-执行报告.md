- 主干先定义 source、collector、transport、processor、store、query、event identity、schema、delivery、backpressure、retention 和 evidence，再分别映射 Elastic、OpenSearch 与 Loki。
- 三条后端均被覆盖但不被描述为无差异替换品；本任务没有选择全局主栈、版本、拓扑、存储、许可证/订阅、资源规格或托管方式。
- Kafka 只作可选缓冲与解耦，不重复阶段 10 的消息系统运维。阶段 06 保有单机日志，阶段 13/14 保有容器与 Kubernetes 日志产生语义，阶段 17 保有指标、告警、追踪和跨信号可观测性。
- 本阶段没有扩展到 SIEM/SOC、完整合规、日志驱动 AIOps、指标/追踪平台或云厂商托管选型。

## 8. 动态兼容与 Promtail EOL 边界

- Elastic Stack 内部组件必须按官方 support matrix 取可证明交集，来源可访问不等于组合已环境验证。
- Filebeat 官方直接输出目标为 Elasticsearch 或 Logstash；不得把现代 Elastic Beats 外推为可直接兼容 OpenSearch。OpenSearch 官方说明新于 Beats OSS 7.12.x 的 Beats 不受支持，兼容 Logstash 与 OpenSearch output plugin 等路径必须在后续阶段门重新核定。
- Promtail 官方已标记于 2026-03-02 EOL，不能作为新部署默认采集器。后续选择门比较 Grafana Alloy、OpenTelemetry Collector、Fluent Bit 或其他受支持客户端；Loki 对第三方客户端的支持责任与自身组件不同。
- 企业日志平台版本组合在软件版本基线中继续为 `phase-gated` 和 `not-yet-verified`；本大纲不把来源核验提升为安装、运行或兼容验证。

## 9. 安全、数据与运维边界

- 日志可能包含凭据、个人数据和业务敏感数据，采集前必须最小化、脱敏并实施最小权限、TLS、认证、授权、租户隔离和审计。
- 高基数 label/field、mapping explosion、重复、丢失、乱序、背压和 retention 成本都进入显式章节或治理边界。
- 平台 snapshot、backup 与 restore 只覆盖平台状态和数据责任，不能冒充业务日志源的备份与恢复能力。
- 本任务没有安装服务、启动进程、发送日志、创建索引或数据流、修改集群、登录平台、使用凭据或操作云资源。
