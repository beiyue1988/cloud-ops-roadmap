## 3. M-R1-01 根因与关闭方式

- **根因：** R1 的 `_aggregate_repository()` 在整个 `_aggregate_repository_checked()` 外捕获 `(OSError, ValueError)`，`main()` 又在整个聚合调用外重复捕获。因此内容规则或跨文件聚合抛出的同类型编程错误会被误转换成脱敏 `IO003`，公开 API 无法暴露真实失败。
- **关闭方式：** `_aggregate_repository()` 的 `try` 现在只覆盖根路径 `resolve`、`exists/is_dir` 和 Markdown 发现/排序；内容读取、单文件规则、Front Matter 与跨文件聚合在 `try` 外执行。`main()` 不再捕获聚合调用。
- **保持行为：** 根路径 `ValueError/OSError` 仍稳定转换为 `.:0: IO003`，CLI 返回 1、无 traceback 且不回显异常原文；根目录缺失仍为 `IO001`，单文件 UTF-8 读取仍为 `IO002`。
- **负向边界：** 从 `validate_placeholders` 注入的 `ValueError`、`OSError` 和额外独立探针中的 `RuntimeError`，均从 `validate_repository()` 与 `main()` 原类型、原消息传播。
- **捕获约束：** 没有 bare `except` 或 `except Exception`；聚合路径 `try` 中没有 `_aggregate_repository_checked`、`validate_*` 或 `_front_matter_diagnostics` 调用。

## 7. 对 R1 执行报告结论的纠正

原 P0-04-R1 执行报告是历史快照，未修改。其“只捕获 `ValueError/OSError`，不会吞掉编程错误”以及“未捕获的编程错误仍按设计向外暴露”的结论过宽：独立复审证明，内容规则抛出的 `ValueError/OSError` 也会被整段聚合捕获并转换成 `IO003`。R1 的新证据只证明了根路径异常被转换，并未证明捕获位置精确。

R2 修复后的准确结论是：只有根路径规范化、根目录状态检查与 Markdown 发现/排序中的 `ValueError/OSError` 转换为脱敏 `IO003`；内容规则和聚合阶段的 `ValueError/OSError/RuntimeError` 不转换、不吞掉，继续向公开调用者传播。README 原有文字描述的是根路径错误范围，与这一精确边界一致，因此保持固定 blob 不变。
