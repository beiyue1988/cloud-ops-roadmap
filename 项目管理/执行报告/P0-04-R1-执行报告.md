## 3. M-01、M-02 与 S-01 根因及关闭方式

### 3.1 M-01：畸形 quoted scalar 被接受

- **根因：** 原 `_strip_scalar()` 只在首尾为同一种引号时去除外层引号，其余文本全部作为 plain scalar 接受，未区分未闭合、首尾不匹配或闭合后尾随内容。
- **关闭方式：** 用受控 `_parse_scalar()` 区分空值、plain、合法成对单/双引号与非法 quoted scalar；不实现 YAML 转义。顶层、列表项和 `versions` 映射值统一调用该分类器；非法结构进入现有结构错误集合并渲染为 `FM008`。
- **脱离实现的观察证据：** 独立探针对顶层第 2 行、列表第 8 行、映射第 16 行分别得到 `FM008`；测试同时覆盖未闭合双引号、未闭合单引号、首尾不匹配、闭合后尾随，以及合法 plain/单引号/双引号/列表/映射正例。

### 3.2 M-02：根路径异常逃逸

- **根因：** 原 `_aggregate_repository()` 在 `Path.resolve()`、`exists/is_dir` 和 Markdown 发现外围没有预期路径/文件系统异常边界；`main()` 直接调用聚合层，`ValueError/OSError` 可逃逸并由脚本入口打印 traceback。
- **关闭方式：** 保留原正常聚合逻辑，在根路径处理边界只捕获 `ValueError` 与 `OSError`，统一返回固定脱敏的 `.:0: IO003 根路径解析或遍历失败`；`main()` 同样保护 `Path` 构造与聚合入口。未使用 bare `except` 或 `except Exception`，编程错误不会被吞掉。
- **脱离实现的观察证据：** `main([chr(0)])` 和注入根路径 `OSError` 均返回 1，stdout 为空，stderr 含 `.:0: IO003`，stdout/stderr 均无 `Traceback`，异常敏感原文未出现；公开 `validate_repository(Path(chr(0)))` 返回唯一脱敏 `IO003`。

### 3.3 S-01：排序测试存在自证

- **根因：** 原测试调用实现自身 `_error_sort_key` 生成期望排序，且没有直接观察同一路径第 2 行与第 10 行的顺序。
- **关闭方式：** 保留两次运行一致性断言，删除所有私有排序函数期望来源；在同一个 Markdown 中制造第 2/10 行 `PH001`，直接断言渲染顺序为 2、10。
- **证据：** 测试文件中 `_error_sort_key` 或以实现私有函数作为 `sorted` key 的命中数为 0；独立探针观察顺序精确为 `2,10`。

## 7. README 与原执行报告结论纠正

README 已把规则范围从 `IO001`–`IO002` 更新为 `IO001`–`IO003`：

- `IO001`：根路径不存在或不是目录；
- `IO002`：单个 Markdown 无法按 UTF-8 读取；
- `IO003`：根路径规范化、目录发现或相关遍历触发预期的 `ValueError/OSError`。

README 同时说明 `IO003` 使用固定脱敏消息，并明确成对 quoted scalar 与复杂引号拒绝边界；仍明确本工具不是完整 YAML、完整 Markdown 或安全审计。

原 [P0-04 执行报告](P0-04-执行报告.md) 是历史快照，未修改。其“读取失败不会产生 traceback”与人工检查 10/10 的结论，证据实际只覆盖单文件非法 UTF-8、缺失根目录和常规 CLI 失败，不能覆盖根路径规范化/发现自身抛出的 `ValueError/OSError`，因此适用范围过宽。P0-04B 独立审查已证明该缺口；本次 R1 以 `IO003`、两个公开 API 探针及无 traceback/脱敏断言提供新的扩展证据。修复后的结论应限定为：批准的根路径 `ValueError/OSError`、根目录缺失/非目录和单文件 UTF-8 读取错误均被稳定转换；未捕获的编程错误仍按设计向外暴露。
