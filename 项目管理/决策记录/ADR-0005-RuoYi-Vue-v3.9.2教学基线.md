# ADR-0005：固定 RuoYi-Vue v3.9.2 教学基线

## 决策元数据

- 编号：ADR-0005
- 标题：固定 RuoYi-Vue v3.9.2 release tag 作为贯穿项目基线
- 状态：`accepted`
- 日期：2026-07-12
- 制定者：主会话；P0-05 执行会话按已批准决定记录
- 影响范围：传统部署、Docker 容器化、Kubernetes 项目、第三方归属与应用依赖核定

## 背景

贯穿项目需要一个可重复获取的真实业务应用。若教程跟随持续变化的 `master`，相同步骤可能在不同时间得到不同代码和依赖；若完整复制第三方源码，又会增加更新、归属和许可证维护成本。官方 Gitee Release 在执行日明确将 `v3.9.2` 指向 commit `0e2d75c`，仓库 LICENSE 为 MIT License。

## 约束与决策问题

- 决策问题：传统项目应固定一个 release 对象，还是持续跟随官方 `master`？
- 产品与学习约束：传统部署到 Kubernetes 的多个阶段必须使用同一业务基线，便于复现、排障和比较演进。
- 技术与环境约束：应用依赖版本仍需在对应阶段门核定；固定 tag 不自动成为全仓库 JDK、MySQL、Redis 或前端工具版本。
- 时间与成本约束：减少上游漂移造成的反复重写，同时不能忽略安全修复和升级成本。
- 权限与治理约束：第三方源码保持 MIT License 与归属；默认不把完整上游源码提交到本仓库；升级必须由替代 ADR 决定。

## 选项 1：固定 v3.9.2 release tag

- 方案：从 `https://gitee.com/y_project/RuoYi-Vue.git` 获取 tag `v3.9.2`，固定 commit `0e2d75c`；教学材料记录上游和 MIT License，默认不提交完整源码。
- 优点：代码对象稳定，可让传统部署、容器化和 Kubernetes 改造共享可复验基线；tag 与 commit 双重标识可发现上游异常移动；第三方边界清晰。
- 缺点：上游后续安全修复和功能不会自动进入；固定版本的依赖可能老化，需要显式评估、补丁或迁移。
- 实施与迁移：项目任务使用固定 tag 的浅克隆并校验 commit；进入传统部署前核定依赖。回退时恢复固定 tag 与已验收配置；升级须由新 ADR 指定新 tag、迁移和验证范围。
- 证据：[v3.9.2 Release](https://gitee.com/y_project/RuoYi-Vue/releases/tag/v3.9.2)、[官方上游](https://gitee.com/y_project/RuoYi-Vue.git)、[MIT LICENSE](https://gitee.com/y_project/RuoYi-Vue/blob/master/LICENSE)，核验日期 2026-07-12。

## 选项 2：跟随上游 master

- 方案：每次项目执行从官方 Gitee `master` 获取当时最新代码，并同步调整依赖、脚本和教程。
- 优点：可以更快获得上游功能、修复和依赖更新；无需单独发起 tag 升级才能看到变化。
- 缺点：同一教程会随时间获取不同 commit，破坏复现性；上游分支、依赖和行为变化会跨多个阶段触发重写与回归，难以稳定保存故障和回退证据。
- 实施与迁移：每次执行记录 commit、重新核定全部依赖并回归所有项目阶段；回退必须保留旧 commit 和构建产物，实质上仍需自行维护版本快照。
- 证据：[官方仓库页](https://gitee.com/y_project/RuoYi-Vue)显示 `master` 为活动分支并独立于 release tag，核验日期 2026-07-12。

## 决定

选择：选项 1。固定官方 Gitee 上游的 `v3.9.2` tag 与 commit `0e2d75c`；源码保持 MIT License 和第三方归属；默认不提交完整第三方源码。应用依赖版本仍为各阶段门输入，不因 tag 固定而自动升级为全局版本。

## 理由

贯穿项目的核心价值是让学习者比较同一业务从传统部署到容器和 Kubernetes 的演进。固定 release tag 能稳定代码输入，并通过 commit 提供更强追溯；跟随 `master` 虽能快速获得变化，却会将上游漂移扩散到所有项目阶段。固定版本接受“修复不会自动进入”的代价，并以安全复审与替代 ADR 管理升级。

## 后果

### 正面后果

- 多阶段项目共享同一可追溯业务代码；
- tag、commit、上游和许可证边界明确；
- 教程可以保存稳定的部署、故障和回退证据。

### 负面后果

- 上游新安全修复不会自动进入，必须监测并评估；
- tag 依赖可能与后续平台版本产生兼容成本；
- 新 tag 迁移需要跨传统、容器和 Kubernetes 项目回归。

### 中性后果

- 获取源码发生在后续项目任务，不在本仓库存储完整第三方代码；
- Nginx、JDK、MySQL、Redis 和前端构建依赖继续由阶段门单独核定。

## 实施与迁移影响

- 实施步骤：P1 前重新确认 tag/commit/许可证；按固定 tag 获取；核定依赖；完成传统部署验证；后续阶段在同一 commit 上演进。
- 文件与系统影响：后续项目说明记录获取命令、commit 校验和第三方归属；任何源码引入必须另有精确授权。
- 兼容与迁移：从 `master` 示例迁移时先固定 `v3.9.2`，重做配置与数据库验证；升级到新 release 时保留旧基线、差异、数据迁移和回滚路径。
- 回退条件：新 tag 迁移失败、关键功能回归或依赖组合不可复现时，恢复 v3.9.2/0e2d75c 与已验收环境；若 v3.9.2 本身出现不可接受安全风险，则停止使用并由新 ADR 选择替代版本。
- 验证与验收：当前只验证官方 release、仓库与许可证事实；环境门必须验证源码 commit、构建、数据库、Redis、前后端运行、发布和回滚。

## 复审触发条件

- v3.9.2 出现关键安全问题、上游删除/移动 tag、仓库不可用或许可证边界变化；
- 阶段门依赖与固定 tag 无法形成受支持且可复现的组合；
- 贯穿项目需求必须使用新功能或新分支；
- 最晚在 P1 传统项目任务进入 `ready` 前复核 tag、commit、依赖与安全状态。
