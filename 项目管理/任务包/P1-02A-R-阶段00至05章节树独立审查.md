## 6. 人工审查标准

### 6.1 双重可用与文件级可签发

- 零基础读者按阶段和前置能连续学习，不依赖隐藏术语；
- 每章只有一个主要目标，能在 30–120 分钟内形成独立正文任务；
- 标题、目标和预定文件足以签发，不包含命令、配置或正文抢写；
- 相邻章节既不机械拆碎同一目标，也不把多个不可独立验收目标塞进一章。

### 6.2 阶段职责与覆盖

- 00：岗位、访问链路、企业 IT、学习方法和实验安全；
- 01：计算机、服务器硬件、操作系统和资源指标；
- 02：网络原理、地址、路由、DNS、HTTP 和基础排查；
- 03：本地虚拟化、云实验环境、终端工具和安全清理；
- 04：Linux 文件、用户、权限、进程、软件包和文本操作；
- 05：启动、systemd、存储、主机网络、时间、内核参数和备份管理。

必须检查跨阶段重复和职责漂移，尤其是 02/05 网络原理与主机配置、03/04 环境与 Linux 操作、04/05 基础服务使用与 systemd 深入、05/06 日志、05/07 参数机制与性能安全、05/08 手工管理与自动化、05/09–10 备份原则与具体服务。

### 6.3 标签、实践和版本边界

- `必须学` 的全部祖先也必须是 `必须学`；唯一 `就业后补学` 不得成为就业主线隐藏前置；
- CP 与阶段一致；LAB-L1/L2 只表达后续锚点引用，不假定实验已经定义或执行；
- VMware 与云 ECS 保持双重可用，不能编码为共同必需环境；
- Kubernetes、GPU、模型与 AI Infra 动态组合不得成为组 A 成立条件；
- VMware、Rocky Linux、RuoYi 等版本声明与软件版本基线、已接受版本门和证据等级一致；
- `source-verified` 不得写成 Rocky Linux 9.8 `environment-verified`。

## 9. 发现分级与建议

- **Critical：** 图或标签闭包错误、安全/版本边界破坏、正式正文越界、固定历史或范围不可信；
- **Important：** 隐藏必要前置、职责重复、明显缺项、不可签发粒度、错误来源声明、报告与当前产品矛盾；
- **Minor：** 不影响连续性和签发的措辞、排序或证据呈现改进；
- **passed：** 无 Critical/Important，全部门禁和范围通过；
- **changes-required：** 至少一个 Critical/Important；
- **blocked：** 固定对象、环境或权威规格冲突使审查无法完成。

发现问题时仍应在唯一报告中完整记录并提交；禁止顺手修产品。最终 `accepted` 只由 Phase 1 主会话决定。

## 10. 固定报告结构与提交

报告必须包含：

1. 角色、日期、固定输入和签发完整哈希；
2. Git 历史、提交范围和工作区证据；
3. 六阶段结构、覆盖合同和聚合统计；
4. 全图、19 条跨阶段边和传递冗余审计；
5. 零基础连续性、粒度和文件级可签发结论；
6. 阶段职责、重复和后续边界结论；
7. 标签、实践锚点、双重可用和版本边界结论；
8. 来源抽查与证据等级限制；
9. 133、默认、partial、catalogs 预期失败、正式正文和 Git 检查的实际输出；
10. Critical/Important/Minor 发现及 `file:line`、影响和最小建议；
11. 风险、未决问题、远程操作确认和状态建议；
12. 明确最终验收权属于 Phase 1 主会话。

| 提交顺序 | 精确范围 | 固定提交信息 |
|---:|---|---|
| 1 | `项目管理/验收记录/P1-02A-阶段章节树独立审查报告.md` | `docs: review group A stage outlines` |

禁止 amend、rebase、reset、改写历史或提交任何其他文件。

## 11. 停止条件

- 当前分支不是 `main`、origin 不符或工作区存在未知修改；
- 固定输入、签发提交或历史产品/报告提交缺失；
- 看板状态不符或其他会话同时写同一报告；
- 133 项、默认、partial、catalogs 预期失败或全图审计不符合固定预期；
- 九类正式非 README 学习 Markdown 不为 0；
- 发现 Critical/Important 或必须修改产品才能继续；
- 权威规格冲突无法消解；
- 需要外部写、凭据、安全控制变更或未授权动态版本选择；
- 审查独立性受损。
