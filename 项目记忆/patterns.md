# 可复用模式

## 单任务、单所有权、独立验收

1. 主会话签发包含精确文件白名单和验证命令的任务包。
2. 执行会话一次只完成一个任务包，记录命令、退出码、关键输出和提交哈希。
3. 主会话先检查范围和 Git 差异，再检查技术、教学、来源与格式证据。
4. 高风险、跨模块或阶段性成果由独立审查会话给出建议，最终状态仍由主会话决定。

## 规格与写作规则分层

1. `AGENTS.md` 管理角色、权限、协作和验收责任。
2. `CODEX-PRD.md`、批准 ADR 与版本基线管理产品范围、架构决策和技术结论。
3. [AI-CODEX-RULES.md](../AI-CODEX-RULES.md) 管理跨内容任务的研究、写作、可视化、安全和验证。
4. `模板/` 管理具体交付物类型的字段与结构。
5. 当前任务包管理本次目标、文件白名单、验证命令和停止条件。
6. 下层规则与上层规格冲突时停止并升级裁决，不在下层文件中自行修订上层基线。

## 安全记录

资源文档只记录公开入口、用途和安全注入方式，不记录可用凭据。任何项目记忆文件都不得保存密码、Token、Cookie、私钥、云 AccessKey 或个人敏感信息。

## 分层版本教学

1. 核心概念尽量保持版本中立，真实版本差异必须标注适用边界。
2. 只维护一套受官方支持且能够复验的主实验环境，动态 patch 从统一版本入口读取。
3. 企业兼容版本用于解释存量环境、组件矩阵和迁移判断，不复制第二套完整课程。
4. EOL 版本只用于历史、技术债务和逐版本升级教学，必须标明风险，不得成为新部署默认线。

## 历史提交空白检查

1. 新差异和当前有效文件继续严格执行 `git diff --check` 与 `git show --check`。
2. 跨阶段审查遇到历史固定提交中的 Markdown 双空格硬换行时，区分当前缺陷、已验收的有意语义和已关闭问题。
3. 已解释的历史语义例外如实记录退出码，不通过改写历史消除证据；只有当前文件或新差异存在未解释空白时才要求返工。
