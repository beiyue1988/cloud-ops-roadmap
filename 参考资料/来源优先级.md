# 来源优先级与证据记录

## 目的

来源按事实所有者和事实类型路由，不以一份简单排名覆盖所有场景。资料被打开不等于结论成立；链接必须直接支持相邻陈述，并记录时点、范围和不能证明的内容。

## 按事实类型选择来源

| 事实类型 | 首选来源 | 交叉核验 | 主要限制 |
|---|---|---|---|
| 标准、协议与格式 | 标准组织、RFC、规范发布者 | 实现项目的官方兼容说明 | 实现可能只支持规范子集 |
| 版本、生命周期、API 与功能 | 上游官方文档、Release、tag、支持页 | 官方仓库与变更日志 | “当前”必须带核验日期和适用分支 |
| 多组件兼容性 | 双方官方兼容矩阵或要求，采用可证明交集 | Release Notes 与后续环境验证 | 单方支持声明不能自动证明另一方或本项目环境 |
| 发行版包、路径与默认行为 | Rocky Linux 或对应发行版官方文档、仓库元数据 | 上游文档并明确发行版差异 | 上游默认值不能直接外推到发行版包 |
| 云产品、实例、库存与价格 | 阿里云官方产品文档、价格页与带日期的控制台观察 | 地域、计费方式、实例条件和价格计算器 | 库存与价格动态变化，必须记录地域、计费方式和时间 |
| 安全公告 | 上游或厂商安全公告、CVE 分配机构 | 发行版安全公告与修复版本 | CVE 存在不等于当前环境受影响，仍需版本和配置判断 |
| 许可证与归属 | 材料仓库中 tag 对应的 LICENSE、NOTICE、版权文件 | SPDX 或许可证官方说明 | 默认分支文件不能无条件替代固定 tag；若 tag 视图受限须记录并交叉核验 |
| 历史与设计动机 | 官方项目历史、设计文档、Release Notes | 经典书籍或经交叉验证的资料 | 回忆、博客或二手叙述不能单独支撑关键时间点 |

## 直接来源与发现线索

搜索摘要、聚合下载站、百科、培训博客、论坛和 AI 生成文本可以用于发现线索，但不是可归档的一手证据。关键动态事实必须重新打开事实所有者的直接页面；第三方文章不能成为版本、生命周期、兼容性、安全、许可证、云库存或价格结论的唯一来源。

不得长篇复制外部内容。使用自己的结构概括直接支持的结论，并保留原始 URL。

## 来源记录的七个字段

每项关键证据至少记录：

1. 被支持的结论；
2. 发布者；
3. 直接 URL；
4. 适用版本或范围；
5. 核验日期；
6. 证据类型；
7. 不能证明的内容。

动态事实必须写核验日期与适用范围。“最新”“当前”“支持”若没有日期和分支边界，不构成可复验记录。

## 兼容性与环境验证

多组件组合使用双方官方支持范围的交集。例如 A 声明支持 B 的某版本，并不自动证明 B 也声明支持 A；只有双方要求能够形成交集，才可标为 `source-verified`。随后仍需在项目基线环境运行，才能取得 `environment-verified`。

## 来源冲突处理

1. 确认来源是否讨论同一产品、版本、日期和适用范围；
2. 优先采用事实所有者的直接来源；
3. 兼容性采用双方明确支持范围的交集；
4. 比较发布日期、更新时间、Release 与文档分支；
5. 记录能够解释的口径差异、结论和限制；
6. 无法解释时停止结论，将组件保持 `phase-gated` 或 `not-yet-verified`，交由主会话裁决。

不得通过选择更方便的第三方来源、猜测版本或模糊措辞绕过冲突。

## 云产品的额外记录

云实例、GPU、显存、地域、库存和价格必须同时记录：核验日期与时间、地域、可用区（适用时）、实例规格、计费方式、购买时长或用量假设、币种/税费口径和控制台可用性。产品文档入口存在只能证明产品范围，不能证明执行时库存或固定价格。

## 许可证的额外记录

许可证与归属优先使用固定 tag 对应的 LICENSE/NOTICE。记录仓库、tag/commit、文件路径、版权所有者与许可证标识；若工具无法解析 tag 文件页，只能使用同一官方仓库对象交叉核验并明确限制，不得由搜索摘要补足。

## 证据声明边界

- 来源页面可访问：仅证明页面在核验时可打开；
- `source-verified`：直接来源支持明确结论；
- 静态检查：只证明格式、链接或规则通过；
- `environment-verified`：明确环境中的实际运行证据；
- 未取得运行证据时使用 `not-yet-verified` 并写最晚验证门。
