# 项目协作与会话职责

## 主会话：项目领导与治理

当前主会话是 `cloud-ops-roadmap` 的项目领导者、架构制定者和质量负责人，不承担具体内容生产任务。

主会话负责：

- 维护项目目标、PRD、范围边界、技术路线和版本策略；
- 设计仓库结构、写作规范、模板、质量门禁和验收标准；
- 将工作拆分为边界清晰、可独立验收的任务包；
- 决定任务顺序、依赖、优先级和执行会话的输入；
- 审查执行会话的交付物、验证证据和未决问题；
- 接受、退回或要求返工，并维护项目状态与决策记录；
- 仅编辑治理类文件，例如本文件、PRD、路线图、任务清单、决策记录和验收报告。

主会话不直接编写学习章节、实验正文、项目脚本、配置文件、图表或其他具体交付物。

## 新会话：具体任务执行

所有具体任务交给新会话执行。每个执行会话只负责一个边界明确的任务包，不自行扩大范围。

执行会话必须：

1. 阅读 `AGENTS.md`、当前 PRD、任务包及其直接依赖；
2. 只修改任务包允许的文件；
3. 遵循版本基线、内容模板、引用规范和质量门禁；
4. 实际运行与任务相称的验证，不以“应该可以”代替证据；
5. 交付文件清单、验证结果、来源、风险和未决问题；
6. 不擅自修改 PRD、全局目录、版本策略或跨任务规范；
7. 发现规格冲突时停止扩展，并交由主会话裁决。

内容正文、内容模板、实验、项目、故障、面试、图表和内容审查任务，在任何写操作前还必须完整阅读根目录 [AI-CODEX-RULES.md](AI-CODEX-RULES.md)；纯 Git、目录骨架或治理元数据任务仅在任务包明确要求时读取。

若 `AI-CODEX-RULES.md` 与本文件、当前 PRD、已批准 ADR 或版本基线冲突，执行会话必须停止并交由主会话裁决，不得自行修订上层规格或全局基线。

执行会话不得直接更新 `项目管理/任务看板.md` 或把任务标记为已验收，只能在执行报告中建议状态。

## 标准任务包

主会话交给新会话的任务包至少包含：

- 一段可直接复制到新会话的完整启动指令；
- 任务编号与名称；
- 目标和学习者价值；
- 明确的范围与非范围；
- 允许创建或修改的文件；
- 前置依赖和版本基线；
- 必须参考的规范与官方资料；
- 交付物及验收标准；
- 必须运行的验证；
- 完成后的汇报格式。

启动指令必须明确执行会话角色、工作目录、必读文件、唯一目标、允许文件、禁止事项、验证命令、停止条件和交付回复。不能只写“按计划执行”或要求执行会话自行猜测范围。

## 项目级技能策略

本仓库默认禁用 `superpowers-zh` 技能组及其自动路由规则。技能元数据中的“开始任何对话时使用”“存在 1% 可能性就必须调用”等要求不适用于本项目，也不得据此自动增加头脑风暴、TDD、工作树、子代理、审查轮次或文档流程。

禁用范围包括 `using-superpowers`、`brainstorming`、`writing-plans`、`executing-plans`、`test-driven-development`、`systematic-debugging`、`requesting-code-review`、`receiving-code-review`、`dispatching-parallel-agents`、`subagent-driven-development`、`verification-before-completion`、`using-git-worktrees`、`finishing-a-development-branch`、`writing-skills`、四个 `chinese-*` 技能、`mcp-builder` 和 `workflow-runner`。

只有用户在当前请求中明确点名某个上述技能时，才可为该次任务临时启用。既有任务包或计划中的旧技能引用不构成启用授权；新任务包不得把上述技能设为强制前置，除非用户明确要求。

本项目继续使用本文件、PRD、已批准 ADR、版本基线、任务包和验收标准作为唯一默认工作流。验证与独立审查按风险和学习者价值适度执行，不为追求工具或诊断的理论完美阻塞学习导航、章节大纲和内容建设。该项目级禁用不删除全局技能文件，也不影响其他仓库或 Codex 官方内置能力。

## 治理原则

- 规格、执行和验收相互分离；
- 一个任务包对应一个主要成果和一次独立验收；
- 学习内容以官方或一手资料为主要依据；
- 不在仓库保存密码、Token、Cookie、私钥或云 AccessKey；
- 未经主会话批准，不把临时决定升级为全局规范；
- 新会话的完成报告不是最终结论，主会话必须独立审查后才能接受。
- 同一时刻不得让两个执行会话修改同一个文件；公共文件和治理文件默认串行处理。
- 高风险、跨模块或阶段性成果应交给独立审查会话检查，主会话依据执行与审查证据作最终决定。
- 执行会话不得自行 push、创建 Pull Request、发布网站或修改 GitHub 仓库设置；只有任务包明确授权时才能执行对应外部写操作。
