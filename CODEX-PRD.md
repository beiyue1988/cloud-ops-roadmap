## 1. 项目定义

### 1.1 项目定位

`cloud-ops-roadmap` 是一套面向零基础学习者的云计算运维成长知识库，兼具三种形态：

- 零基础课程教材：提供清晰的学习顺序、前置知识、解释和检查点；
- 企业培训手册：强调可复现实验、生产规范、故障处理和工程表达；
- 长期维护知识库：支持按技术查询、版本更新、内容复用和后续发布为网站或 PDF。

主成长路径为：

> Linux 运维 → 云计算运维 → 云原生运维 → DevOps → SRE 基础 → AI Infra 入门

AIOps 与 AI Infra 必须分开：

- **AI Infra** 关注 GPU、CUDA、容器、Kubernetes GPU 调度和大模型推理服务的建设与运维；
- **AIOps** 关注使用 AI 辅助告警、日志分析、故障定位、巡检和自动化处置。

### 1.2 目标用户

主要用户：

- 不理解服务器、IP、Linux 等概念的完全零基础学习者。

兼顾用户：

- 具备日常电脑使用能力但没有 Linux 基础的人；
- 有开发经验、希望转向运维或 DevOps 的人；
- 希望补齐体系的初级运维工程师。

所有正文默认按主要用户编写，不假设读者已经理解英文缩写、命令行、网络或编程。

### 1.3 最终能力目标

完整学习并完成核心实践后，学习者应能够：

1. 安装、初始化、维护和排查 Rocky Linux 服务器；
2. 理解企业网络、存储、权限、安全、日志和性能的基础机制；
3. 使用 Shell、Python、Git、Ansible 和 IaC 工具完成基础自动化；
4. 部署并维护 Nginx、Java 应用、MySQL、Redis 等常见服务；
5. 在 VMware 和阿里云中规划并管理计算、网络、存储和安全资源；
6. 将传统应用改造为 Docker 容器，并使用 Kubernetes 进行编排；
7. 建设基础 CI/CD、集中日志、指标监控、告警和链路追踪能力；
8. 理解 GPU 服务器、CUDA 软件栈和大模型推理服务，完成入门级 AI Infra 实践；
9. 使用分层、证据驱动的方法处理故障、变更、复盘和容量问题；
10. 在面试中用“原理 + 操作 + 排障 + 生产权衡”表达真实工程能力。

### 1.4 非目标

本项目不承诺：

- 在固定三个月内学完整套知识或保证就业；
- 覆盖 Linux 内核开发、Kubernetes 源码开发或数据库内核开发；
- 替代算法、机器学习数学和大模型训练课程；
- 提供任何厂商认证的应试路线；
- 把所有工具都讲到专家或架构师深度；
- 用篇幅堆砌历史、命令参数或过时技术。

## 2. 核心内容原则

### 2.1 完整知识库与就业路线分离

仓库保存完整的初级至中级知识范围，不因“三个月就业路线”删除内容。时间约束只影响学习路径的优先级：

- **完整成长路线**：覆盖项目规划的全部主题；
- **三个月就业主线**：标记必须学、建议学、就业后补学；
- **专题路线**：为转开发、云原生提升、AI Infra 入门等用户提供不同入口。

学习时间只写“建议投入”，不承诺掌握时长。

### 2.2 实践与理论比例

全仓库目标比例约为：

- 70% 实践、实验、项目、排障、规范与工程案例；
- 30% 背景、概念、原理、演进与架构思想。

该比例是阶段级和仓库级目标，不机械要求每个小章节都达到 7:3。

### 2.3 背景与历史的取舍

技术背景不能缺失，但必须服务于理解今天的设计：

- Linux、虚拟化、云计算、容器、Kubernetes、可观测性、GPU 和 AI Infra 应讲清关键演进；
- Nginx、Redis、Ansible 等工具只说明其产生的工程问题与重要转折；
- `grep`、`chmod`、`curl` 等单个命令不写发展史；
- 不罗列创始人、年份或版本流水账，除非它直接解释兼容性、淘汰原因或架构变化。

### 2.4 初级至中级深度边界

每项技术至少覆盖“是什么、为什么、怎么用、如何验证、如何排障、生产中如何取舍”。

原则深度示例：

- Linux：理解进程、内存、文件系统、权限、systemd、日志与性能，不进入内核开发；
- Docker：理解 image、container、registry、namespace、cgroup、存储和网络，不分析源码；
- Kubernetes：理解声明式 API、控制循环、调度、网络、存储、安全和升级，不分析 Scheduler 源码；
- MySQL：覆盖备份恢复、复制、索引和常见性能问题，不进入存储引擎源码；
- AI Infra：覆盖部署、资源、兼容性和运维，不推导 Transformer 数学或训练算法。

### 3.3 三个月就业主线

三个月路线不是删减版仓库，而是优先级导航：

1. 第 1–2 周：运维认知、实验环境、必要的计算机和网络基础；
2. 第 3–6 周：Linux 基础、系统管理、Linux 日志、性能和安全；
3. 第 7–9 周：Shell、Nginx、Java 运行、MySQL、Redis；
4. 第 10 周：若依传统部署、发布、备份和故障演练；
5. 第 11 周：Git、Ansible、阿里云基础；
6. 第 12 周：Docker 基础、综合复习、项目表达和面试模拟。

Kubernetes、完整 CI/CD、企业日志平台、可观测性、AI Infra 和 SRE 标记为“就业后持续学习”，但保留在完整知识库。

`04-每日每周学习模板.md` 提供可调整的学习节奏：每日按“理解 → 操作 → 验证 → 复盘”组织，每周按“新知识 → 实验 → 综合任务 → 查漏补缺”组织。模板只帮助安排投入，不设置强制完成日，也不因学习速度不同裁剪知识。

## 5. 文档模型

### 5.1 章节颗粒度

- 一个可独立学习和检查的知识点对应一个 Markdown 文件；
- 单章建议投入通常为 30–120 分钟；
- 大主题必须拆分，例如“Linux 权限”“sudo”“ACL”分别成章；
- 命令速查表、实验、故障案例和面试题不与知识正文混成超长文件；
- 每章只承担一个主要教学目标。

### 5.2 统一 Front Matter

机器可读键使用英文，值和正文以中文为主：

```yaml
---
title: Linux 文件权限基础
chapter: "04.06"
stage: Linux 基础
type: principle
difficulty: beginner
prerequisites:
  - 04.05-Linux用户与用户组
objectives:
  - 解释 Owner、Group、Other 与 rwx
  - 使用 chmod 和 chown 完成权限配置
study_time: 60-90分钟
practice_level: Level-1
keywords:
  - Linux
  - permission
  - chmod
versions:
  rocky_linux: "9.8"
status: draft
last_updated: 2026-07-10
---
```

允许的 `type`：

- `concept`：概念或认知；
- `principle`：原理或协议；
- `service`：软件服务；
- `platform`：平台；
- `tool`：工具；
- `language`：编程或脚本语言；
- `cloud-service`：云产品；
- `hardware`：硬件与 AI Infra；
- `methodology`：流程与方法论。

允许的 `status`：`draft`、`reviewed`、`verified`、`deprecated`。

### 5.3 自适应章节模板

不允许用同一个僵化模板套所有知识。所有章节共享以下核心骨架：

1. 本章目标与前置知识；
2. 技术定位：是什么、解决什么问题、位于哪里；
3. 核心概念或关键规则；
4. 原理、流程或使用模型；
5. 实际场景或实践；
6. 验证与常见问题；
7. 学习检查；
8. 参考资料。

根据 `type` 增加模块：

| 类型 | 必选扩展 | 按需扩展 |
|---|---|---|
| concept / principle | 背景、机制、流程、抓包或观测 | 演进、对比、故障树 |
| service | 架构、安装、配置、生命周期、日志、备份、排障 | 高可用、性能、安全 |
| platform | 整体架构、组件、资源模型、部署方式、日常运维 | 扩缩容、升级、多租户 |
| tool | 工作模式、安装配置、基础操作、流程集成 | 插件、扩展、高级自动化 |
| language | 语法、编程模型、规范、测试、项目实践 | 性能和第三方库 |
| cloud-service | 产品定位、资源模型、控制台与 API、权限、计费意识 | HA、成本优化、跨地域 |
| hardware | 组成、指标、软件栈、观测工具、兼容性 | 拓扑、容量、性能调优 |
| methodology | 原则、角色、流程、输入输出、案例 | 成熟度模型和反模式 |

“背景与演进”“安装部署”“高可用”“性能优化”等均为按类型选择的模块。不存在内容时不创建空标题。

### 5.4 术语与表达

- 首次出现时写中文解释和英文关键词，例如“服务级别目标（Service Level Objective, SLO）”；
- 后续使用行业常见缩写，例如 SLO、RTO、CNI；
- 命令、配置键、路径、API 对象和代码保持原始英文；
- 不生硬翻译 Docker、Kubernetes、Pod、Deployment 等专有名词；
- 类比只用于建立直觉，紧随准确原理和适用边界；
- 不用“显而易见”“很简单”等否定新手感受的措辞。

## 6. 辅助学习体系

每个阶段根据内容选择以下辅助材料：

- 技术定位图和知识前置关系图；
- 术语表和英文关键词；
- 常见误区与反例；
- 命令或 API 速查表；
- 理论检查题、动手任务和阶段挑战；
- 面试高频问题、回答结构和追问方向；
- 官方文档、RFC、标准、书籍和深入阅读；
- 淘汰技术或技术演进说明，仅在有助于选型和兼容性理解时加入。

辅助材料服务于理解和复习，不复制正文。

## 7. 可视化规范

### 7.1 工具分工

- Mermaid：流程图、时序图、状态图、依赖关系和小型数据流；
- Draw.io：系统架构、网络拓扑、云资源边界、Kubernetes 集群、日志/监控平台和 AI Infra 架构；
- 表格：精确字段映射、版本对比、命令对比和方案权衡；
- 禁止以 ASCII 图作为正式架构图；目录树和终端输出可以使用等宽文本。

### 7.2 是否画图的判断

图必须让关系比文字更容易理解。以下内容优先画图：

- 三个及以上组件的依赖、数据流或控制流；
- 分层、边界、拓扑、主从、集群或高可用关系；
- 多步骤请求、启动、发布、故障或恢复流程；
- 状态变化、时序交互、资源生命周期；
- 需要沿链路排查的问题。

单个命令、一个事实、一两步操作和短列表不强制画图。核心技术应覆盖定位、架构、流程、排障四类认知，但可由多章共同完成，不强制每个小章节放四张图。

### 7.3 Draw.io 规范

- 源文件保存为 `.drawio`，文档默认引用导出的 `.svg`；
- 源文件与导出图同名，例如 `Kubernetes-整体架构.drawio` 与 `Kubernetes-整体架构.svg`；
- 每张图只表达一个主要主题，标题说明“读完应理解什么”；
- 流向默认从左到右，分层默认从上到下；
- 颜色不得作为唯一编码，同时使用分组标题、边框或图标；
- 固定语义：蓝色为基础设施，绿色为应用/计算，紫色为平台/编排，橙色为数据，红色仅用于故障与风险；
- 同类节点保持相同形状、圆角、线宽、字号和箭头；
- 正文解释放在图外，图内节点使用短标签；
- 图中必须标明信任边界、网络区域或数据方向（适用时）。

### 7.4 Mermaid 规范

- 节点 ID 使用 ASCII，显示标签使用中文；
- 一张图建议不超过 12 个主要节点；
- 复杂图改用 Draw.io；
- 流程判断必须标明“是/否”或明确条件；
- 代码块下方用一小段文字解释关键路径，不能只给图不解释。

### 7.5 图编号

编号格式：`FIG-领域-类型-序号`，例如：

- `FIG-LINUX-FLOW-001`：Linux 启动流程；
- `FIG-K8S-ARCH-001`：Kubernetes 整体架构；
- `FIG-OBS-DATA-001`：可观测性数据流。

## 8. 实验体系

### 8.1 实验等级

| 等级 | 目标 | 示例 |
|---|---|---|
| Level 1 | 会操作和验证单个知识点 | 用户权限、Nginx 安装、systemctl 管理 |
| Level 2 | 能组合多个知识点完成部署 | LNMP、MySQL 备份恢复、Ansible 批量配置 |
| Level 3 | 能维护并处理故障 | 磁盘满、502、复制中断、Pod 异常、日志堆积 |
| Level 4 | 能设计方案并解释取舍 | 高可用、灾备、可观测平台、GPU 推理服务架构 |

### 8.2 实验模板

每个实验必须包含：

1. 实验目标与完成标准；
2. 前置知识；
3. 环境、拓扑、主机名、IP 和版本；
4. 风险提示、快照点和回滚方法；
5. 操作步骤与必要解释；
6. 每个阶段的验证命令及预期现象；
7. 至少一个常见错误或故障注入（适用时）；
8. 清理或恢复步骤；
9. 实验总结；
10. 扩展练习。

实验不得依赖未说明的账号、付费资源、私有镜像或隐藏文件。涉及阿里云计费资源时，先写成本提示和释放步骤。

### 8.3 可复现要求

- 命令必须在规定版本环境中实际验证；
- 配置文件提供完整可运行版本，不只给零散片段；
- 所有变量、路径、端口、网段和凭据来源必须说明；
- 不在仓库保存密码、Token、Cookie、私钥和云 AccessKey；
- 破坏性命令必须解释影响并给出可验证的目标范围；
- 实验完成后应能从快照或清理步骤恢复到已知状态。

### 11.2 版本维护策略

- 正文引用 `版本记录/软件版本基线.md`，避免重复写死补丁版本；

## 15. 多会话项目治理

### 15.1 角色分离

本项目采用“领导会话制定与验收，执行会话生产交付物”的工作模式：

- **主会话**：项目领导者、规格制定者、任务拆分者和最终验收者；
- **执行会话**：领取一个任务包，完成被授权的具体文件并提供验证证据；
- **审查会话**：对高风险、跨模块或阶段性成果进行独立技术与教学审查；
- **用户**：决定产品方向、重大范围变化、公开许可和涉及真实成本或外部发布的事项。

主会话可以编辑 `AGENTS.md`、PRD、计划、任务包、任务看板、决策记录和验收报告，但不直接编写学习正文、实验、脚本、配置或图表。执行会话不得修改全局规格，发现冲突时必须提交主会话裁决。

本 PRD 规定项目“做什么”；[AI-CODEX-RULES.md](AI-CODEX-RULES.md) 规定跨内容任务“怎么研究、写作、可视化、保证安全和提供验证证据”，但不重新定义项目范围、技术基线或会话权限。

项目官方 GitHub 远程仓库固定为：

```text
origin = https://github.com/beiyue1988/cloud-ops-roadmap.git
```

P0-01 负责在初始化本地 Git 后配置并验证 `origin`。关联远程不等于授权 push；push、Pull Request、发布和 GitHub 仓库设置变更必须由后续任务包单独授权。

### 15.2 任务状态

任务状态使用以下固定集合：

| 状态 | 含义 | 状态变更责任人 |
|---|---|---|
| `planned` | 已进入路线图但依赖尚未满足 | 主会话 |
| `ready` | 任务包完整，可以交给新会话 | 主会话 |
| `assigned` | 已确定执行会话 | 主会话 |
| `in-progress` | 执行会话正在工作 | 主会话根据执行反馈更新 |
| `submitted` | 交付物和执行报告已提交 | 主会话 |
| `under-review` | 正在进行独立审查或验收 | 主会话 |
| `rework` | 未达到验收标准，需要返工 | 主会话 |
| `accepted` | 已通过验收并纳入项目基线 | 主会话 |
| `blocked` | 存在明确且无法在任务范围内解除的阻塞 | 主会话 |
| `cancelled` | 因范围或优先级变化而取消 | 主会话 |

执行会话只提交结果和状态建议，不直接编辑 `任务看板.md`。任务看板是唯一权威状态源。

### 15.3 任务包门禁

任务只有在同时具备以下信息时才能标记为 `ready`：

1. 唯一任务编号、名称和所属 Phase；
2. 一个主要目标及其学习者价值；
3. 明确的范围和非范围；
4. 精确的允许修改文件列表；
5. 前置依赖、版本基线和必读文件；
6. 官方资料要求；
7. 逐项交付物；
8. 可操作的验收标准；
9. 必须运行的验证命令与预期结果；
10. 固定的执行报告格式。
11. 一段可以直接复制到新会话的完整启动指令，明确角色、工作目录、必读文件、唯一目标、允许文件、禁止事项、验证、停止条件和回复格式。

缺少任何一项时，任务保持 `planned`，不得交给执行会话自行补全规格。

### 15.4 文件所有权与并发

- 同一时刻不得有两个执行会话修改同一个文件；
- 治理、模板、版本基线和目录结构任务默认串行；
- 内容任务只有在章节树、模板和版本基线已 `accepted` 后才能并行；
- 可并行任务必须具有互不重叠的文件清单和稳定的上游依赖；
- 跨模块公共文件由主会话单独开任务，不附带在普通章节任务中；
- 任务出现范围外修改需求时，执行会话只记录建议，不顺手修改。

### 15.5 提交与验收

执行会话提交时必须提供：

- 创建和修改的文件；
- 每项验收标准的对应证据；
- 实际运行的命令、退出码和关键输出；
- 使用的官方来源及核验日期；
- 已知限制、风险和未解决问题；
- 是否建议进入独立审查。

主会话按以下顺序验收：规格范围 → 文件边界 → 技术证据 → 教学质量 → 来源与版本 → 链接和格式 → 集成影响。高风险任务必须先交给独立审查会话；执行会话的“完成”声明不能代替主会话验收。

### 15.6 返工与变更控制

- 未通过验收时，主会话创建带有明确差距和复验命令的返工任务包；
- 返工优先交给原执行会话，无法继续时再开新的执行会话；
- 影响多个 Phase、目录结构、版本基线或完成定义的修改必须先更新 PRD 或正式决策记录；
- 普通事实纠错可以直接进入修复任务，不需要扩大项目范围；
- 已 `accepted` 的规范被替代时，旧决策移动到 `deprecated` 记录并保留迁移说明。

### 15.7 决策门

以下事项不阻塞 Phase 0，但必须在对应节点前由用户或主会话完成决策：

| 决策 | 最晚决策节点 | 默认处理 |

## 17. 质量门禁

### 17.1 内容门禁

- 读者无需猜测前置概念；
- 重要术语首次出现有中文解释与英文关键词；
- 背景解释技术为何存在，但不写无关历史；
- 命令不是孤立清单，每个关键操作有目的、影响和验证；
- 理论与生产场景建立连接；
- Linux 单机日志与企业集中日志明确分章；
- AI Infra 与 AIOps 明确分章；
- 没有空章节、重复正文和无来源的版本断言。

### 17.2 技术门禁

- 所有实验在目标版本验证；
- 关键命令和配置保留验证记录；
- 版本、API 和产品功能链接官方依据；
- 安全、破坏性操作和付费资源有显著提示；
- 脚本通过对应 Lint 或基础测试；
- Mermaid 可渲染，Draw.io 源文件可打开，SVG 导出与源一致；
- Markdown 内部链接、标题锚点和图片引用有效。

### 17.3 教学门禁

- 每章有可检查的学习目标；
- 每章末尾有自测或动手检查；
- 每阶段有综合挑战和完成标准；
- 三个月路线能链接到就业主线所需的全部章节、实验和项目；
- 项目从传统部署演进到容器、Kubernetes 和可观测性；
- 面试内容基于正文和已完成实验，不形成独立背题孤岛。

### 17.4 完成定义

单个章节只有同时满足以下条件才能标记 `verified`：

1. Front Matter 完整；
2. 技术内容已核验；
3. 命令或实验已在基线环境验证；
4. 图和链接可用；
5. 有学习检查；
6. 至少一次内容审查通过；
7. 没有占位符、未解释缩写或已知错误。

## 18. 持续维护规则

- 高价值稳定规则写入 `AGENTS.md`；
- 详细决策、版本原因和更新经验写入项目记忆文件；
- 每完成一个阶段更新技术索引和学习路线；
- 每月检查外部链接和高变动版本；
- 每季度审查版本基线、弃用 API、云产品变化和安全建议；
- 过时内容标记 `deprecated`，写迁移方向后再删除；
- 发现错误优先修正文和关联实验，再更新故障或面试内容；
- 不保存任何真实 Secret 或个人敏感信息。
